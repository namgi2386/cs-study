## 번들 최적화 기법

> 번들 최적화는 결국 캐시되지 않은 cold start 상태일때 가장 큰 의미를 가진다.
> Next.js의 경우 JS, CSS, 정적파일을 자동 캐싱한다. 단, HTML은 캐싱하지 않는다. response로 받아오는 data는 메모리 캐싱한다.

### manualChunks

- 청크를 묶는 기준을 설정할 수 있다.
  - vite.config.ts 에서 수정가능
  - vendor-react, vendor-query, ui모듈, app모듈 등으로 구분 하는 방식
- 어떤 기준으로 묶을까
  - 변경 빈도와 크기를 기준으로 나눠라.
  - 변경 빈도가 낮은 라이브러리 React, lodash를 vendor 청크로 분리
  - 재사용되지만, 빈도가 낮은 라이브러리 components 청크로 분리
  - 유틸/컴포넌트/페이지 로 분리하거나
  - 라이브러리별로 분리하거나
  - 폴더(페이지)별로 분리하거나
  - 프로젝트에 적절하게 분리하자.
- 청크를 어느 크기로 줄일까
  - HTTP2부터는 병렬 다운로드를 지원하기에 chunk를 잘게 나누는게 유리하다. 단, 청크가 5KB 미만이면 HTTP 헤더/핸드셰이크 오버헤드가 더 크다
  - 사용하지 않는 dependency를 담은 chunk는 제거하자

### React.lazy 활용한 동적 import 전환

- 장점
  - 초기 번들 크기 감소 (필요할때 필요한 chunk만 import)
  - 초기 로딩 시간 단축
  - 단, 동적import페이지 진입시에는 필연적 로딩 발생
- 비교
  - 동적import는 직접 로딩/에러 처리 등 해야한다.
  - React.lazy는 간결하게 해줌. 그래서 씀. 안써도 됨.
- 예시 상황
  - 메인 -> A page (B component) 구조 일때,
  - 동적 import 없이 구현
    - 메인페이지에 입장시, A페이지chunk와 B컴포넌트chunk를 동시에 다운로드
  - 동적 import 사용 (A페이지 내에서 동적 import로 B컴포넌트 불러오도록 함)
    - 메인페이지 입장시, A페이지chunk는 다운로드
    - B컴포넌트chunk는 다운로드 안함
    - 사용자가 A페이지에 입장했을때에 B컴포넌트chunk를 다운로드
    - 즉, 메인페이지 입장 사용자는 B컴포넌트chunk를 다운로드 하지 않아도 된다.
- 참고사항
  - eact.lazy는 import('./파일').default만 인식하므로, named export 페이지는 default export로 변환해서 감싸야 동적 로드가 가능

### 단순 동적 import 와 React.lazy 코드 비교

- 동적 import : `const temp = import('./temp')`

```js
const [Button , setButton] = useState(null);

useEffect(()=> {
  import('./Button').then(c => setButton(c.default));
}, []);

return Button ? <Button/> : <Loading/>;
```

- lazy 활용 : `const temp = React.lazy(() => import('./temp'))`

```js
const Button = React.lazy(() => import('./Button'));

return (
  <Suspense fallback={<Loading/>}>
    <Button/>
  </Suspense>
) 
```
