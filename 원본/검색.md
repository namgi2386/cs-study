## 1. 대기업 검색 엔진 분석

### Google

| 계층 | 기술 |
|------|------|
| 크롤링 | Googlebot (분산 크롤러) → 수백억 페이지 |
| 인덱싱 | 역색인(Inverted Index) + Bigtable/Spanner |
| 1차 랭킹 | BM25 계열 어휘 매칭 |
| 링크 분석 | **PageRank** — 링크를 "투표"로 환산, 권위 전파 |
| 의미 이해 | **BERT**(2019) → **MUM**(2021, 멀티모달 1000배 강력) |
| 최종 랭킹 | **LTR(Learning to Rank)** — 200+ 시그널을 ML 모델로 통합 |
| 최신 | AI Overviews: RAG 방식으로 LLM이 결과 요약 생성 |

> 📖 **용어 설명**
> - **크롤링(Crawling)**: 웹 로봇(봇)이 인터넷의 웹 페이지를 자동으로 돌아다니며 내용을 수집하는 과정. 도서관 사서가 책장을 하나씩 훑으며 목록을 만드는 것과 같다.
> - **Googlebot**: 구글의 크롤러 이름. 수백 대의 서버에서 동시에 실행되어 매일 수십억 개 페이지를 방문한다.
> - **분산 크롤러**: 한 대의 컴퓨터가 아니라 수천 대의 컴퓨터가 동시에 나눠서 크롤링하는 방식. 혼자 도서관 전체를 돌지 않고 수천 명이 구역을 나눠 동시에 정리하는 것.
> - **역색인(Inverted Index)**: 책 뒤에 있는 "찾아보기(색인)"와 동일한 개념. 일반 색인은 "책 → 단어" 순서인데, 역색인은 "단어 → 그 단어가 나오는 문서 목록"으로 **역방향** 매핑. 검색 엔진의 핵심 자료구조 (3섹션에서 상세 설명).
> - **Bigtable / Spanner**: 구글이 자체 개발한 초대형 분산 데이터베이스. 수백 PB(페타바이트, 1PB = 100만 GB)의 인덱스 데이터를 저장하는 데 사용. 일반 MySQL 같은 DB로는 이 규모를 감당 불가.
> - **BM25**: 단어가 문서에 얼마나 잘 맞는지 점수를 계산하는 수학 공식 (4섹션에서 상세 설명).
> - **어휘 매칭(Lexical Matching)**: 사용자가 입력한 단어가 문서에 그대로 들어있는지 글자 단위로 비교하는 방식. "자동차"를 검색하면 "자동차"라는 단어가 있는 문서만 찾음.
> - **PageRank**: 구글 창업자 래리 페이지가 만든 알고리즘. 다른 페이지에서 링크를 많이 받을수록 중요한 페이지로 평가. 학술 논문에서 "인용을 많이 받은 논문이 더 중요하다"는 개념과 동일 (4섹션에서 상세 설명).
> - **BERT(Bidirectional Encoder Representations from Transformers)**: 구글이 2018년 공개한 AI 언어 모델. "배" 라는 단어가 '복부'인지 '과일'인지 '선박'인지 앞뒤 문맥을 보고 구분하는 능력 보유. 검색 쿼리의 실제 의도를 파악하는 데 사용.
> - **MUM(Multitask Unified Model)**: BERT보다 1,000배 강력한 구글의 AI 모델. 텍스트뿐 아니라 이미지, 동영상 등 여러 형태의 정보를 동시에 이해하는 '멀티모달' 능력 보유.
> - **멀티모달(Multimodal)**: 텍스트, 이미지, 음성, 동영상 등 여러 종류의 데이터를 동시에 처리하는 능력. "이 사진 속 산의 이름이 뭐야?"처럼 이미지와 텍스트를 함께 이해하는 것.
> - **LTR(Learning to Rank, 학습 기반 순위 결정)**: 머신러닝을 사용해 검색 결과의 순위를 결정하는 기법. BM25 점수, PageRank, 클릭률 등 수백 가지 신호를 ML 모델이 조합해 최적 순위를 출력.
> - **ML 모델(Machine Learning Model)**: 데이터를 학습하여 패턴을 찾고 예측/판단을 하는 수학적 모델. 여기서는 "어떤 페이지가 이 검색어에 더 관련 있는가"를 판단.
> - **시그널(Signal)**: 검색 순위를 결정할 때 참고하는 모든 데이터 포인트. 페이지 로딩 속도, 클릭률, 체류 시간, 모바일 최적화 여부 등 200가지 이상.
> - **파이프라인(Pipeline)**: 여러 처리 단계를 순서대로 연결한 구조. 물이 파이프를 통해 단계별로 처리되는 것에서 유래. "크롤링 → 인덱싱 → 랭킹" 순으로 흐르는 구조.
> - **AI Overviews**: 구글 검색 결과 상단에 AI가 자동으로 생성한 요약 답변을 보여주는 기능.
> - **RAG(Retrieval-Augmented Generation)**: 검색으로 관련 문서를 먼저 찾은 후, AI가 그 내용을 바탕으로 답변을 생성하는 방식. AI가 "기억"하는 것 외에 최신 정보도 참고 가능 (8섹션에서 상세 설명).
> - **LLM(Large Language Model, 거대 언어 모델)**: GPT, Gemini처럼 수천억 개의 텍스트로 학습된 초대형 AI 모델. 문장을 읽고 쓰는 능력이 매우 뛰어남.

**핵심 인사이트**: Google은 단일 알고리즘이 아니라 크롤링 → 인덱싱 → 다단계 랭킹의 **파이프라인**이다.

### YouTube

두 단계 딥러닝 아키텍처 (논문: "Deep Neural Networks for YouTube Recommendations", 2016)

```
[수십억 동영상]
      ↓
 Candidate Generation (수천 개로 축소)
 - 사용자 시청 이력 임베딩
 - 협업 필터링 (유사 사용자 클러스터링)
 - 콘텐츠 메타데이터 (제목, 태그, 카테고리)
      ↓
 Ranking (수십 개로 정렬)
 - 시청 시간, 클릭률, 좋아요, 댓글 등 engagement 신호
 - 개인화 특징 벡터
 - 신선도(freshness) 가중치
```

> 📖 **용어 설명**
> - **딥러닝 아키텍처(Deep Learning Architecture)**: 여러 층(layer)으로 쌓인 신경망으로 구성된 AI 모델 구조. "깊은(deep)" 이유는 층이 많기 때문.
> - **Candidate Generation(후보 생성)**: 수십억 개의 전체 동영상에서 "이 사용자에게 보여줄 가능성이 있는" 수천 개의 후보를 추려내는 1단계. 촘촘하게 거르지 않고 빠르게 걸러냄.
> - **임베딩(Embedding)**: 텍스트, 영상, 사용자 등 복잡한 데이터를 수백~수천 개의 숫자 배열(벡터)로 변환하는 것. "자동차"와 "승용차"는 의미가 비슷하므로 비슷한 숫자 배열을 가짐. 의미를 수학적으로 표현하는 방법.
> - **협업 필터링(Collaborative Filtering)**: "나와 비슷한 시청 패턴을 가진 다른 사용자들이 본 영상을 나도 좋아할 것이다"라는 원리. 넷플릭스 추천의 기본 원리.
> - **클러스터링(Clustering)**: 비슷한 특성을 가진 데이터끼리 자동으로 그룹을 짓는 과정. 여기서는 "나와 비슷한 취향의 사용자 그룹"을 만드는 데 사용.
> - **콘텐츠 메타데이터(Content Metadata)**: 동영상 자체가 아닌 동영상에 대한 설명 데이터. 제목, 태그, 카테고리, 업로드 날짜, 채널 이름 등.
> - **Engagement 신호**: 사용자가 콘텐츠에 얼마나 반응했는지 나타내는 데이터. 시청 시간, 좋아요, 댓글, 공유, 클릭률 등이 포함됨.
> - **개인화 특징 벡터(Personalization Feature Vector)**: 특정 사용자의 취향을 숫자 배열로 표현한 것. 이 숫자를 보면 이 사람이 어떤 영상을 좋아하는지 알 수 있음.
> - **신선도(Freshness)**: 최근에 업로드된 콘텐츠를 약간 더 높이 평가하는 가중치. 오래된 뉴스보다 최신 뉴스를 위에 띄우는 것과 동일.
> - **Transformer 기반 순차 모델**: "어제 A 영상 → 오늘 B 영상 → 다음은 C 영상을 볼 것이다"처럼 시청 순서(시퀀스)에서 패턴을 학습하는 AI 모델. GPT도 같은 Transformer 구조.

- 하루 **800억 개** 이상의 신호를 처리
- 검색 쿼리 + 추천 모두 동일한 임베딩 공간 활용
- 최근에는 **Transformer 기반 순차 모델**로 시청 이력 모델링

### GitHub Code Search

**역사적 전환**: Elasticsearch → 자체 개발 **Blackbird** (Rust)

| 문제 | 해결책 |
|------|--------|
| ES는 일반 텍스트 검색 — 부분 문자열 매칭 불가 | **Trigram Index** |
| 2억+ 레포 실시간 인덱싱 불가 | Rust 기반 완전 재작성 |
| 정규식 검색 | Trigram → Regex 변환으로 후보 축소 후 정확 매칭 |

**Trigram 인덱싱 원리**:
- `"limits"` → `lim`, `imi`, `mit`, `its` 3-gram으로 분해
- 각 trigram → 해당 파일 목록의 역색인
- 쿼리 시 trigram 교집합으로 후보 파일 추출 → 정규식 정확 매칭

> 📖 **용어 설명**
> - **Elasticsearch**: 오픈소스 검색 엔진 소프트웨어. Apache Lucene을 기반으로 만들어졌으며 설치만 하면 강력한 전문 검색(Full-Text Search)이 가능. 일반 텍스트 검색에 최적화.
> - **Blackbird**: GitHub이 2022년 Rust 언어로 직접 개발한 코드 검색 엔진. 코드 검색에 특화된 기능(부분 문자열 검색, 정규식 등)이 Elasticsearch보다 뛰어남.
> - **Rust**: 메모리 안전성과 초고속 실행 속도가 특징인 시스템 프로그래밍 언어. C/C++ 수준의 성능을 내면서도 메모리 버그가 없어 대규모 인프라 개발에 인기.
> - **부분 문자열 매칭(Substring Matching)**: "getUser" 를 검색할 때 "getUserById", "getUserInfo" 등 중간에 해당 문자열이 포함된 결과를 찾는 것. 코드 검색의 핵심 기능이지만 일반 역색인으로는 구현이 어려움.
> - **Trigram(트라이그램)**: 연속된 3개의 문자 조합. "limits" → `lim`, `imi`, `mit`, `its` 처럼 한 칸씩 밀면서 3글자씩 추출. 이 trigram들의 역색인을 만들면 어떤 부분 문자열도 검색 가능.
> - **정규식(Regular Expression, Regex)**: 문자열 패턴을 표현하는 언어. `get.*User`처럼 와일드카드 패턴으로 코드를 검색할 수 있음. 예: `get.*By(Id|Name)$`.
> - **교집합(Intersection)**: 두 집합에서 공통으로 속하는 원소만 추출하는 연산. "limits"를 검색하면 `lim`이 있는 파일 AND `imi`이 있는 파일 AND `mit`이 있는 파일의 교집합만 후보로 추림.

### Elasticsearch / Apache Lucene (업계 표준)

오픈소스 검색의 사실상 표준. Twitter, Wikipedia, Netflix, Airbnb 등이 사용.

**세그먼트 구조**:
```
Index
 └─ Shard (수평 분할)
     └─ Segment (불변, immutable)
         ├─ Inverted Index (텀 → 문서 목록)
         ├─ Term Dictionary (정렬된 유니크 텀 사전)
         ├─ Posting List (문서 ID, TF, 위치)
         └─ Doc Values (정렬/집계용 컬럼 스토어)
```

세그먼트는 **불변(immutable)** → 수정 시 새 세그먼트 생성 후 백그라운드 병합(merge)

> 📖 **용어 설명**
> - **Apache Lucene**: Elasticsearch 내부에서 실제 검색을 담당하는 핵심 라이브러리. Elasticsearch는 Lucene을 감싸서 분산처리, REST API 등을 추가한 제품.
> - **Index(인덱스)**: 검색 대상 문서의 모음. 관계형 DB의 "테이블"과 유사한 개념. 하나의 Elasticsearch 클러스터에 여러 인덱스를 만들 수 있음.
> - **Shard(샤드)**: 인덱스를 여러 조각으로 나눈 것. 도서관의 책을 여러 서가에 나눠서 보관하는 것과 같음. 각 샤드는 독립적으로 검색 가능하며 서로 다른 서버에 배치해 병렬 처리.
> - **수평 분할(Horizontal Partitioning)**: 데이터를 여러 서버에 나눠 저장하는 방식. 수직 분할(더 좋은 서버 한 대)의 반대 개념으로, 서버를 계속 추가해 용량/성능 확장 가능.
> - **Segment(세그먼트)**: 샤드 내부의 작은 독립 인덱스 조각. 새 문서가 추가될 때마다 새 세그먼트가 생성됨. 세그먼트가 여러 개 있으면 검색 시 모든 세그먼트를 병렬로 검색 후 결과를 합침.
> - **불변(Immutable)**: 한 번 만들어지면 수정할 수 없다는 의미. 세그먼트는 불변이기 때문에 문서를 수정하면 기존 세그먼트의 문서를 "삭제 표시"하고 새 세그먼트에 수정된 문서를 추가하는 방식으로 처리.
> - **Term Dictionary(텀 사전)**: 인덱스에 저장된 모든 단어를 정렬된 상태로 저장한 사전. 단어 검색 시 이진 탐색으로 O(log n) 빠르게 찾음.
> - **Posting List(포스팅 리스트)**: 특정 단어가 등장하는 문서들의 목록. "apple"이라는 단어가 1번, 4번, 7번 문서에 있다면 포스팅 리스트는 [1, 4, 7].
> - **Doc Values**: 문서의 필드 값을 컬럼 형태로 따로 저장한 것. 날짜 정렬, 가격 범위 필터 같은 집계/정렬 연산에 사용. 역색인은 "단어 → 문서"이지만, Doc Values는 "문서 → 값" 방향.
> - **컬럼 스토어(Column Store)**: 데이터를 행(row) 단위가 아닌 열(column) 단위로 저장하는 방식. 특정 필드의 전체 값을 빠르게 읽을 수 있어 집계/정렬에 유리.
> - **백그라운드 병합(Background Merge)**: 세그먼트가 너무 많아지면 검색 속도가 느려지므로, 서비스에 영향을 주지 않도록 백그라운드에서 작은 세그먼트들을 합쳐 큰 세그먼트로 만드는 작업.

## 2. 검색의 기본 개념

### 검색의 두 가지 근본 문제

1. **어디서 찾을 것인가** — 탐색 공간 축소 (인덱싱, 필터링)
2. **무엇이 더 관련있는가** — 관련도 순위 (랭킹, 스코어링)

> 📖 **용어 설명**
> - **인덱싱(Indexing)**: 나중에 빠르게 검색할 수 있도록 문서를 미리 가공해서 저장하는 과정. 책을 읽기 전에 뒤에 있는 "찾아보기" 목록을 먼저 만드는 작업.
> - **필터링(Filtering)**: 검색 전에 특정 조건에 맞지 않는 문서를 미리 제외하는 과정. "2024년 이후" 같은 날짜 조건으로 대상을 줄이는 것.
> - **랭킹(Ranking)**: 검색 결과를 관련도 순서대로 정렬하는 과정. 단순히 "포함/비포함"이 아니라 "얼마나 잘 맞는지" 순서를 매김.
> - **스코어링(Scoring)**: 각 문서에 관련도 점수(숫자)를 계산하는 과정. 이 점수로 랭킹을 결정.

### 검색 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| **Exact Match** | 완전 일치 | DB WHERE 쿼리 |
| **Full-Text Search** | 형태소 분석 + 어휘 매칭 | Elasticsearch |
| **Prefix/Wildcard** | 접두사, 패턴 매칭 | 자동완성 |
| **Fuzzy Search** | 오타 허용 (편집 거리) | 스펠 교정 |
| **Semantic Search** | 의미 기반 벡터 유사도 | AI 검색 |
| **Faceted Search** | 필터 기반 탐색 | 쇼핑 검색 |

> 📖 **용어 설명**
> - **Exact Match(완전 일치 검색)**: 입력한 문자열과 완전히 동일한 것만 찾는 방식. SQL의 `WHERE name = '홍길동'` 이 대표적.
> - **Full-Text Search(전문 검색)**: 문서 전체 내용을 대상으로 자연어 검색. 형태소 분석으로 단어를 분해 후 역색인으로 검색. "삼성 스마트폰 후기" → "삼성", "스마트폰", "후기" 각각 검색.
> - **형태소 분석(Morphological Analysis)**: 단어를 의미 있는 최소 단위(형태소)로 분해하는 과정. "먹었습니다" → "먹(동사 어근)" + "었(과거)" + "습니다(경어)". 한국어 검색의 핵심.
> - **Prefix Search(접두사 검색)**: 단어의 앞부분만 입력해도 검색되는 방식. "appl"을 입력하면 "apple", "application", "apply" 등이 나오는 자동완성.
> - **Wildcard Search(와일드카드 검색)**: `*`(모든 문자) 또는 `?`(단일 문자) 같은 특수 기호로 패턴을 지정하는 검색. `app*` → apple, apply, application 모두 매칭.
> - **Fuzzy Search(퍼지 검색)**: 오타나 유사 철자를 허용하는 검색. "googl" 입력 시 "google"을 찾아주는 것. 기준은 편집 거리(Edit Distance).
> - **편집 거리(Edit Distance / Levenshtein Distance)**: 두 문자열이 얼마나 다른지 수치화. "cat" → "car"는 1글자만 바꾸면 되므로 편집 거리 1. 퍼지 검색에서 허용 오차 기준으로 사용.
> - **Semantic Search(의미론적 검색)**: 단어 자체가 아닌 의미를 기반으로 검색. "강아지"와 "반려견"은 글자는 다르지만 의미가 같으므로 매칭. AI 임베딩 벡터의 유사도로 판단.
> - **벡터 유사도(Vector Similarity)**: 두 벡터(숫자 배열) 간의 유사한 정도. 코사인 유사도, 유클리드 거리 등으로 측정 (6섹션에서 상세 설명).
> - **Faceted Search(패싯 검색)**: 여러 필터를 동시에 적용해 범위를 좁혀가는 검색. 쇼핑몰에서 "브랜드: 삼성 + 가격: 50만원 이하 + 색상: 검정"처럼 조건을 추가하며 탐색.

### 텍스트 분석 파이프라인 (Tokenization → Normalization)

```
원문: "Running Faster Than Expected"
  ↓ Tokenizer    → ["Running", "Faster", "Than", "Expected"]
  ↓ Lowercase    → ["running", "faster", "than", "expected"]
  ↓ Stop Words   → ["running", "faster", "expected"]
  ↓ Stemmer      → ["run", "fast", "expect"]
  ↓ 역색인에 저장
```

> 📖 **용어 설명**
> - **Tokenization(토크나이제이션)**: 문장을 개별 단어나 의미 단위(토큰)로 나누는 과정. "I love cats" → ["I", "love", "cats"]. 공백이나 구두점 기준으로 나눔.
> - **Normalization(정규화)**: 서로 다른 형태의 동일한 표현을 하나의 표준 형태로 통일하는 과정. 대소문자, 시제, 복수형 등을 통일.
> - **Lowercase(소문자 변환)**: "Apple", "APPLE", "apple"을 모두 "apple"로 통일. 대소문자 구분 없이 검색 가능하게 함.
> - **Stop Words(불용어)**: 검색에 의미가 없는 단어들. 영어의 "the", "a", "is", 한국어의 "은", "는", "이", "가" 등. 이런 단어는 인덱스에서 제외해 공간을 절약하고 노이즈를 줄임.
> - **Stemming(스테밍 / 어간 추출)**: 단어의 어미를 제거해 어간(root) 형태로 변환. "running", "runs", "ran" → 모두 "run"으로 통일. 규칙 기반이라 빠르지만 "better" → "bet"처럼 잘못 추출되기도 함.
> - **Lemmatization(레마타이제이션 / 표제어 추출)**: 사전 기반으로 단어의 원형(표제어)을 찾아 변환. "better" → "good", "went" → "go"처럼 문법적으로 올바른 원형 반환. Stemming보다 정확하지만 사전이 필요해 느림.
> - **형태소 분석기(Morphological Analyzer)**: 한국어, 일본어처럼 단어 경계가 불명확한 언어에서 문장을 의미 단위로 분해하는 도구. "아버지가방에들어가신다" → "아버지/가방/에/들어가/신다"처럼 분해. Mecab, Nori, Komoran이 대표적.

- **Stemming**: 어간 추출 (running → run), 규칙 기반
- **Lemmatization**: 표제어 추출 (better → good), 사전 기반, 더 정확
- **한국어**: 형태소 분석기 필수 (Mecab, Nori, Komoran)

## 3. 핵심 자료구조

### 역색인 (Inverted Index)

검색 엔진의 핵심. "단어 → 문서 목록" 매핑.

```
Term Dict     Posting List
─────────────────────────────────────────
"apple"   →  [doc1(tf:3, pos:2,7,12), doc4(tf:1, pos:5)]
"banana"  →  [doc2(tf:2, pos:1,8),   doc3(tf:5, pos:3,...)]
"cherry"  →  [doc1(tf:1, pos:9),     doc3(tf:2, pos:1,6)]
```

> 📖 **용어 설명**
> - **Term Dict(텀 사전)**: 인덱스에 저장된 모든 단어를 알파벳/가나다 순으로 정렬해 저장한 사전. 특정 단어가 존재하는지 빠르게 확인 가능.
> - **Posting List(포스팅 리스트)**: 특정 단어가 등장하는 모든 문서의 목록. 각 항목에는 문서 번호(doc ID), 등장 횟수(TF), 위치(pos) 정보가 포함.
> - **TF(Term Frequency, 단어 빈도)**: 특정 단어가 한 문서에 등장하는 횟수. `tf:3`이면 그 문서에 3번 등장했다는 뜻.
> - **pos(Position, 위치)**: 단어가 문서의 몇 번째 위치에 있는지 저장. `pos:2,7,12`이면 2번째, 7번째, 12번째 단어 위치에 등장. "apple pie"처럼 단어의 순서가 중요한 구문 검색(phrase query)에 사용.
> - **구문 검색(Phrase Query)**: "apple pie"처럼 단어들이 특정 순서로 연속해서 나타나는 경우를 검색. 단순 역색인으로는 불가능하고 위치 정보가 필요.
> - **doc ID(문서 ID)**: 각 문서에 부여된 고유 번호. 포스팅 리스트는 doc ID를 오름차순으로 정렬해 저장 → AND 연산 시 두 리스트를 동시에 훑어가며 O(n) 시간에 교집합 계산 가능.
> - **O(n) merge(선형 시간 병합)**: 두 정렬된 포스팅 리스트를 병합할 때, 두 포인터를 앞에서부터 순서대로 이동하면 전체 크기에 비례하는 O(n) 시간에 처리 가능. 정렬이 안 되어 있으면 O(n²).

- **TF (Term Frequency)**: 문서 내 등장 횟수
- **Position**: 구문 검색(phrase query)용 위치 정보
- 포스팅 리스트는 **doc ID 오름차순 정렬** → AND 연산이 O(n) merge

### Trie (접두사 트리)

자동완성, 타이핑 제안에 사용.
- 공통 접두사 공유로 공간 절약
- `apple`, `apply`, `apt` → 같은 `ap` 노드에서 분기
- 실제로는 **Radix Trie** (압축 트라이) 사용

> 📖 **용어 설명**
> - **Trie(트라이)**: 문자열을 한 글자씩 나눠서 트리(나무) 형태로 저장하는 자료구조. 루트 노드에서 시작해 글자를 하나씩 따라가면 단어를 찾을 수 있음. 자동완성 기능의 기본 구조.
> - **공통 접두사 공유**: "apple"과 "apply"는 앞 3글자 "app"가 같으므로 같은 노드를 공유. 별도 저장 대비 메모리 절약.
> - **접두사(Prefix)**: 단어의 앞부분. "apple"의 접두사는 "a", "ap", "app", "appl", "apple".
> - **분기(Branch)**: 트리에서 경로가 나뉘는 지점. "app" 노드에서 "apple"은 "l" 방향, "apply"는 "l" 방향(공유) 후 "y"로, "apt"는 "t" 방향으로 분기.
> - **Radix Trie(라딕스 트라이 / 압축 트라이)**: 연속된 단일 노드를 하나로 압축한 트라이. "apple" 저장 시 a-p-p-l-e 5개 노드가 아닌 "apple" 1개 노드로 압축해 공간 절약. Patricia Trie라고도 불림.

### Finite State Transducer (FST)

Lucene의 Term Dictionary 구현체.
- Trie보다 훨씬 메모리 효율적
- 공통 접미사도 공유 (DAG 구조)
- Term → 오프셋 매핑을 메모리에 올려 O(1) 조회

> 📖 **용어 설명**
> - **FST(Finite State Transducer, 유한 상태 변환기)**: Trie를 더 압축한 자료구조. Trie는 공통 접두사만 공유하지만, FST는 공통 **접미사**도 공유해 메모리를 극단적으로 절약. Lucene이 Term Dictionary를 메모리에 올릴 때 사용.
> - **접미사(Suffix)**: 단어의 뒷부분. "apply"와 "supply"의 공통 접미사는 "pply".
> - **DAG(Directed Acyclic Graph, 방향성 비순환 그래프)**: 한 방향으로만 이동 가능하고 순환이 없는 그래프. FST는 공통 접미사를 하나의 노드로 공유하므로 트리가 아닌 DAG 구조.
> - **오프셋(Offset)**: 파일이나 배열에서 특정 위치까지의 거리(바이트 수). FST는 "단어 → 파일 내 위치(오프셋)"를 저장해, 실제 포스팅 리스트가 저장된 곳으로 바로 이동 가능.
> - **O(1) 조회**: 데이터 크기와 관계없이 항상 일정한 시간에 조회. 해시맵의 조회가 대표적. FST를 메모리에 올려두면 단어 → 오프셋 변환이 O(1)에 가능.

### Skip List / B-Tree

- 포스팅 리스트의 **Skip Pointer**: AND 연산 시 건너뛰기로 O(n) → 빠르게
- 범위 쿼리는 B-Tree 계열 (날짜, 숫자 필터)

> 📖 **용어 설명**
> - **Skip List(스킵 리스트)**: 정렬된 연결 리스트에 여러 레벨의 "지름길(포인터)"을 추가한 자료구조. 1→2→3→...→1000으로 하나씩 이동하는 대신, 1→100→500→700처럼 건너뛰어 탐색 시간 단축.
> - **Skip Pointer(스킵 포인터)**: 포스팅 리스트 중간중간에 넣어두는 지름길 참조. 예를 들어 "apple"의 포스팅 리스트에서 doc ID 500을 찾을 때, doc ID 1부터 순서대로 가지 않고 스킵 포인터로 500 근처로 점프. AND 연산 속도 향상에 핵심.
> - **AND 연산**: "apple AND banana" 검색 시 두 단어가 모두 있는 문서를 찾는 연산. 두 포스팅 리스트의 교집합.
> - **B-Tree(B-트리)**: 데이터베이스 인덱스의 표준 자료구조. 데이터를 정렬된 상태로 유지하며, "100 이상 200 미만" 같은 범위 쿼리(Range Query)를 효율적으로 처리. MySQL, PostgreSQL의 인덱스가 B-Tree 기반.
> - **범위 쿼리(Range Query)**: "가격이 10만~20만원" 또는 "날짜가 2024년 이후" 처럼 특정 범위 내의 값을 찾는 쿼리.

## 4. 랭킹 알고리즘

### TF-IDF

가장 기본적인 관련도 스코어.

```
TF(t, d)  = 문서 d에서 단어 t의 등장 횟수 / 문서 d의 전체 단어 수
IDF(t)    = log(전체 문서 수 / t를 포함한 문서 수)
TF-IDF    = TF × IDF
```

> 📖 **용어 설명**
> - **TF(Term Frequency, 단어 빈도)**: 특정 단어가 한 문서에 몇 번 등장하는지의 비율. "apple"이 100단어 문서에 5번 등장하면 TF = 5/100 = 0.05. 많이 등장할수록 그 문서가 해당 단어와 관련 있다고 봄.
> - **IDF(Inverse Document Frequency, 역문서 빈도)**: 전체 문서 중 특정 단어가 얼마나 드물게 등장하는지. "the"처럼 모든 문서에 있는 단어는 IDF가 낮고(거의 0), "photosynthesis"처럼 적은 문서에만 있는 단어는 IDF가 높음. log를 사용해 급격한 차이를 완화.
> - **log(로그)**: 지수를 역산하는 수학 함수. 큰 수의 차이를 완화해 줌. 1,000,000 문서와 100,000 문서의 IDF 차이를 log 없이 계산하면 너무 크기 때문에 log를 사용해 1~4 사이의 값으로 압축.
> - **포화(Saturation)**: 단어가 많이 등장할수록 점수가 계속 오르지 않고 일정 수준에서 수렴하는 현상. TF-IDF는 포화를 고려하지 않아 단어를 100번 쓴 문서가 10번 쓴 문서보다 10배 높은 점수를 받는 문제 존재.

- IDF: 희귀한 단어일수록 가중치 높음 ("the"는 낮음, "photosynthesis"는 높음)
- 한계: 문서 길이 미보정, 포화(saturation) 미고려

### BM25 (Best Matching 25)

TF-IDF의 공식 개선판. **현재도 대부분의 검색 엔진 기본 스코어러**.

```
BM25(q, d) = Σ IDF(t) × [TF(t,d) × (k1 + 1)] / [TF(t,d) + k1 × (1 - b + b × |d|/avgdl)]

k1 ≈ 1.2~2.0  : TF 포화 제어 (크면 TF 반영 강화)
b  = 0.75     : 문서 길이 정규화 강도
```

> 📖 **용어 설명**
> - **BM25(Best Matching 25)**: 1994년에 정립된 TF-IDF 개선 공식. 이름의 "25"는 연구 중 25번째 버전이라는 의미. 30년이 지난 지금도 Elasticsearch, Solr, 루씬의 기본 랭킹 공식.
> - **Σ(시그마)**: 합산 기호. 검색 쿼리의 모든 단어(t)에 대해 BM25 점수를 계산하고 더하라는 의미.
> - **k1 파라미터**: TF의 포화 속도를 조절. k1이 클수록 단어가 많이 등장할수록 점수가 계속 오름. k1=0이면 TF 무시. 일반적으로 1.2~2.0 사이 값 사용.
> - **b 파라미터**: 문서 길이 정규화의 강도 조절. b=0이면 길이 무시, b=1이면 완전 정규화. 0.75가 일반적 기본값.
> - **|d|(문서 길이)**: 해당 문서의 전체 단어 수.
> - **avgdl(Average Document Length, 평균 문서 길이)**: 전체 인덱스에 있는 모든 문서의 평균 단어 수. 이 값으로 나누어 "이 문서가 평균보다 긴지 짧은지" 판단.
> - **길이 정규화(Length Normalization)**: 긴 문서는 단어를 많이 포함할 수밖에 없어 자연히 TF가 높아지므로, 문서 길이를 감안해 점수를 조정하는 것. "apple"이 100번 등장하는 1000단어 문서 vs 10번 등장하는 100단어 문서 → 사실상 동일한 비율이므로 비슷한 점수를 줘야 함.

- **포화**: 동일 단어가 10번 나오든 100번 나오든 효과 수렴
- **길이 정규화**: 긴 문서가 단순히 길어서 높은 스코어 받는 것 방지

### PageRank

링크 구조 기반 권위도(Authority) 계산.

```
PR(A) = (1-d) + d × Σ [PR(B) / OutLinks(B)]

d = 0.85  : 감쇠 계수 (랜덤 워커가 링크를 따를 확률)
```

> 📖 **용어 설명**
> - **PageRank**: 구글 공동 창업자 래리 페이지(Larry Page)의 이름을 딴 알고리즘. "좋은 페이지는 좋은 페이지로부터 링크를 받는다"는 원리. 학술 논문의 인용 횟수 분석에서 영감을 얻음.
> - **권위도(Authority)**: 해당 페이지가 얼마나 신뢰할 수 있고 중요한 페이지인지를 나타내는 점수. 뉴욕타임스나 네이처 저널처럼 많은 곳에서 참조하는 페이지는 권위도가 높음.
> - **OutLinks(아웃링크 수)**: 해당 페이지에서 다른 페이지로 나가는 링크의 수. A 페이지가 10개 링크를 갖고 있다면 자신의 PageRank를 10개 페이지에 1/10씩 나눠줌. 링크를 남발하면 효과가 희석됨.
> - **d(감쇠 계수, Damping Factor)**: 인터넷을 무작위로 클릭하는 가상의 사용자가 "링크를 따라갈 확률"(0.85)과 "아무 페이지나 랜덤 방문할 확률"(0.15)을 모델링. d=0.85는 85%는 링크를 따라가고, 15%는 완전 다른 페이지로 점프한다는 의미.
> - **Power Iteration(거듭제곱 반복법)**: PageRank를 계산하는 방법. 처음에는 모든 페이지에 동일한 점수를 주고, 공식을 반복 적용하면 점수가 점점 수렴(안정화)됨. 보통 수십~수백 번 반복.
> - **수렴(Convergence)**: 반복 계산에서 값의 변화가 무시할 수준으로 작아져 안정된 상태가 되는 것.
> - **TrustRank**: PageRank의 스팸 방지 변형. 구글이 수동으로 "신뢰할 수 있는 씨앗 페이지(예: 주요 대학, 뉴스 사이트)"를 선정하고, 이 페이지에서 가까울수록 신뢰도가 높다고 판단. 스팸 사이트가 서로 링크를 걸어 PageRank를 인위로 높이는 것을 방지.

- 수렴할 때까지 반복 계산 (Power Iteration)
- 스팸 방지를 위해 **TrustRank** (신뢰 씨앗에서 전파) 병행

### Learning to Rank (LTR)

**현대 검색 랭킹의 핵심**. 다수의 시그널을 ML 모델로 통합.

| 방식 | 학습 단위 | 예시 |
|------|---------|------|
| Pointwise | 단일 문서 | 회귀/분류 |
| Pairwise | 문서 쌍 | RankNet, LambdaMART |
| Listwise | 전체 순위 | ListNet, LambdaRank |

> 📖 **용어 설명**
> - **LTR(Learning to Rank, 학습 기반 순위 결정)**: BM25, PageRank, 클릭률, 페이지 속도 등 수백 가지 시그널을 ML 모델이 학습해 최적의 순위를 결정. "단일 공식"이 아닌 "학습된 조합"으로 랭킹.
> - **Pointwise(포인트와이즈)**: 각 문서에 독립적으로 점수(관련도 0~5 등)를 예측하는 방식. "이 문서는 쿼리와 4점짜리 관련도"처럼 절대 점수 학습. 가장 단순하지만 다른 문서와의 상대적 순위를 고려하지 않음.
> - **Pairwise(페어와이즈)**: 두 문서를 비교해 "A가 B보다 더 관련 있다/없다"를 학습. "A > B인가?"를 계속 학습하며 순위를 결정. 현재 가장 많이 사용.
> - **Listwise(리스트와이즈)**: 전체 결과 목록을 한 번에 보고 최적 순위를 학습. 가장 정교하지만 계산 비용이 큼.
> - **RankNet**: Microsoft가 개발한 Pairwise LTR 모델. 신경망을 사용해 두 문서의 순위를 학습.
> - **LambdaMART**: RankNet + MART(Gradient Boosted Trees) 결합. Bing, 야후 등에서 사용. Google 내부 시스템도 유사한 GBT 계열 사용.
> - **Gradient Boosted Trees(GBT, 그레이디언트 부스팅 트리)**: 여러 개의 약한 결정 트리를 순서대로 학습해 이전 모델의 오차를 보완하는 앙상블 방법. XGBoost, LightGBM이 대표적.
> - **CTR(Click-Through Rate, 클릭률)**: 검색 결과에서 특정 링크가 노출된 횟수 대비 실제 클릭 횟수 비율. CTR이 높으면 사용자가 이 결과가 유용하다고 판단했다는 신호.
> - **신선도(Freshness)**: 최근에 작성/수정된 콘텐츠에 가산점을 주는 것. 뉴스 검색에서 오래된 기사보다 최신 기사를 위에 배치하는 근거.

- 실제 Google은 **LambdaMART** (Gradient Boosted Trees) 계열 사용
- 특징: BM25 스코어, PageRank, CTR, 신선도, 사용자 체류 시간 등

## 5. 분산 검색 아키텍처

### 수평 분할: Sharding

```
Query → Scatter  → Shard 0 (doc 0~999)
                → Shard 1 (doc 1000~1999)
                → Shard 2 (doc 2000~2999)
     ← Gather   ← 각 shard 상위 K개 결과 병합 → 최종 재랭킹
```

> 📖 **용어 설명**
> - **Sharding(샤딩)**: 거대한 데이터를 여러 서버에 나눠 저장하는 기법. 도서관의 책을 A~G, H~N, O~Z처럼 알파벳별로 다른 서가에 나눠 보관하는 것과 유사. 각 서버가 부분적인 데이터만 담당해 병렬 처리 가능.
> - **Scatter(스캐터)**: 쿼리를 모든 샤드에 동시에 뿌리는 과정. "이 질문을 모든 담당자에게 동시에 물어보는" 것.
> - **Gather(개더)**: 각 샤드에서 받은 결과를 하나로 모으는 과정. 각 담당자가 보낸 답변을 취합하는 것.
> - **재랭킹(Re-ranking)**: 각 샤드에서 받은 상위 K개 결과를 다시 합쳐 최종 순위를 결정하는 과정. 샤드별로 계산한 점수 기준이 다를 수 있어 최종 통합 후 다시 정렬 필요.
> - **문서 기반 샤딩**: 문서를 균등하게 여러 샤드에 분배. 어떤 문서가 어느 샤드에 있는지 예측하기 어렵지만 균형 잡힘. Elasticsearch 기본 방식.
> - **텀 기반 샤딩**: 특정 단어는 항상 특정 샤드에만 존재. "apple" 관련 문서는 모두 샤드1에. 단어 검색 시 해당 샤드만 조회하면 되어 빠르지만, 인기 단어가 있는 샤드에 부하 쏠림.

- **문서 기반 샤딩**: 문서를 균등 분배 (Elasticsearch 기본)
- **텀 기반 샤딩**: 특정 텀이 특정 샤드에만 존재 → 더 빠르나 불균형 위험

### 복제 & 고가용성

- Primary shard + Replica shard
- 읽기는 replica로 분산 → 처리량 증가
- Primary 장애 시 replica 자동 승격

> 📖 **용어 설명**
> - **Primary Shard(프라이머리 샤드)**: 원본 데이터를 가지고 있는 샤드. 쓰기(write) 요청은 반드시 프라이머리 샤드에서 처리.
> - **Replica Shard(레플리카 샤드)**: 프라이머리 샤드의 복사본. 읽기 요청을 분담해 처리량(throughput) 향상. 프라이머리 샤드가 있는 서버가 다운되면 자동으로 새 프라이머리로 승격.
> - **고가용성(High Availability, HA)**: 서버 일부가 장애를 겪어도 서비스가 중단되지 않는 능력. 복제(Replication)가 고가용성의 핵심 수단.
> - **자동 승격(Auto-promotion)**: 원본(Primary) 서버가 죽었을 때 복제본(Replica)이 자동으로 원본 역할을 맡아 서비스 중단 없이 운영되는 것.

### Near Real-Time (NRT) 인덱싱

```
문서 추가
 → In-memory buffer
 → Refresh (기본 1초): 새 세그먼트 생성, 검색 가능
 → Flush: 디스크에 영구 저장 (fsync)
 → Merge: 작은 세그먼트 병합 (백그라운드)
```

> 📖 **용어 설명**
> - **NRT(Near Real-Time, 준실시간)**: 데이터가 추가되고 거의 즉시(수초 이내) 검색에 반영되는 것. 완전한 실시간(0ms)은 아니지만 사용자가 체감하기엔 즉각적.
> - **In-memory buffer(메모리 버퍼)**: 문서를 디스크에 쓰기 전에 메모리에 임시 저장하는 공간. 디스크 쓰기는 메모리보다 수천 배 느리므로 먼저 메모리에 모아뒀다가 한 번에 씀.
> - **Refresh(리프레시)**: 메모리 버퍼의 내용으로 새 세그먼트를 만들어 검색 가능한 상태로 만드는 과정. 기본 1초마다 실행. Refresh 전까지는 새 문서가 검색되지 않음.
> - **Flush(플러시)**: 메모리의 내용을 디스크에 영구적으로 저장하는 과정. 서버가 꺼져도 데이터가 유지됨. Refresh보다 훨씬 덜 자주 실행됨.
> - **fsync(파일 동기화)**: 운영체제의 파일 쓰기 캐시를 비워 실제 디스크에 기록하는 시스템 콜. 이 작업 이후에야 서버가 꺼져도 데이터가 안전하게 보존됨.
> - **Merge(병합)**: 작은 세그먼트들을 합쳐 큰 세그먼트로 만드는 백그라운드 작업. 세그먼트가 많을수록 검색 시 확인해야 할 조각이 많아 느려지므로 주기적으로 병합 필요.

세그먼트가 많아질수록 검색 느려짐 → 백그라운드 merge 필수

### 글로벌 스케일: Google Bigtable / Spanner

- 웹 인덱스 전체는 수백 PB — 단일 Elasticsearch로 불가
- Google: 커스텀 분산 파일시스템(GFS) + Bigtable 위에 자체 인덱서
- **Caffeine** (실시간 인덱싱 시스템): 페이지 크롤 후 수초 내 검색 반영

> 📖 **용어 설명**
> - **PB(페타바이트)**: 1PB = 1,000TB = 100만 GB. 구글의 웹 인덱스는 수백 PB 규모로, 단일 Elasticsearch 클러스터(보통 수십 TB 수준)로는 감당 불가.
> - **GFS(Google File System)**: 구글이 수천 대의 서버를 하나의 거대한 파일 시스템처럼 사용하기 위해 만든 분산 파일시스템. 2003년 논문으로 공개돼 Hadoop HDFS의 영감이 됨.
> - **Caffeine**: 구글의 연속적 웹 크롤링 및 인덱싱 시스템. 기존 시스템은 배치(batch) 방식이라 며칠 걸렸지만, Caffeine은 스트리밍 방식으로 수초 내 검색 반영 가능. 2010년부터 사용.

## 6. 벡터 검색 & ANN

### 왜 벡터 검색인가

BM25는 **어휘 불일치(Vocabulary Mismatch)** 문제 존재:
- 쿼리 "자동차 수리" ≠ 문서 "차량 정비" → BM25는 0점
- 벡터 임베딩은 의미적 유사도를 수치로 표현

> 📖 **용어 설명**
> - **어휘 불일치(Vocabulary Mismatch)**: 사용자가 쓴 단어와 문서에 쓰인 단어가 달라서 관련 있는 문서를 찾지 못하는 문제. "자동차"로 검색하면 "차량"만 쓴 문서는 못 찾음. BM25의 가장 큰 한계.
> - **의미적 유사도(Semantic Similarity)**: 단어가 달라도 의미가 같거나 가까운 정도. "강아지"와 "개"는 의미적 유사도가 매우 높음.

### 임베딩 & 코사인 유사도

```python
# 문서/쿼리를 고차원 벡터로 변환
q_vec = encoder("자동차 수리")      # [0.2, -0.5, 0.8, ...]  (768~1536 차원)
d_vec = encoder("차량 정비 방법")   # [0.19, -0.48, 0.77, ...]

similarity = cosine_similarity(q_vec, d_vec)  # ≈ 0.95
```

> 📖 **용어 설명**
> - **임베딩(Embedding)**: 텍스트, 이미지 등 복잡한 데이터를 숫자 배열(벡터)로 변환하는 것. AI 모델(BERT 등)이 문장의 의미를 수백~수천 개의 숫자로 압축. 의미가 비슷한 문장은 비슷한 숫자 배열을 가짐.
> - **벡터(Vector)**: 방향과 크기를 가진 숫자 배열. 여기서는 단순히 숫자들의 목록. [0.2, -0.5, 0.8] 처럼 표현.
> - **차원(Dimension)**: 벡터를 구성하는 숫자의 개수. 768차원이면 768개의 숫자로 이루어진 벡터. 차원이 높을수록 더 복잡한 의미를 표현 가능.
> - **Encoder(인코더)**: 텍스트를 벡터로 변환하는 AI 모델. BERT, Sentence-BERT, OpenAI의 text-embedding-ada 등이 대표적.
> - **코사인 유사도(Cosine Similarity)**: 두 벡터가 이루는 각도의 코사인 값. 같은 방향이면 1, 반대 방향이면 -1, 무관하면 0. 두 벡터의 길이가 달라도 방향만 비교하므로 텍스트 유사도 측정에 적합. 0.95면 매우 유사.

### KNN vs ANN

- **KNN (Exact)**: 모든 벡터와 비교 → O(n×d), 수억 개면 불가
- **ANN (Approximate)**: 근사 최근접 이웃, 정확도 약간 희생 → 실용적

> 📖 **용어 설명**
> - **KNN(K-Nearest Neighbor, K 최근접 이웃)**: 쿼리 벡터와 데이터베이스의 모든 벡터를 하나씩 비교해 가장 가까운 K개를 찾는 방법. 완전히 정확하지만 데이터가 많을수록 선형적으로 느려짐.
> - **O(n×d)**: 문서 n개, 각 벡터 차원 d개일 때 비교 횟수. 1억 문서 × 768차원이면 768억 번 계산 → 실용적으로 너무 느림.
> - **ANN(Approximate Nearest Neighbor, 근사 최근접 이웃)**: 정확한 최근접 이웃을 찾지 않고 "충분히 가까운" 이웃을 빠르게 찾는 방법. 99% 정확도로 100배 빠른 검색이 가능. 실제 프로덕션에서는 ANN이 필수.

### HNSW (Hierarchical Navigable Small World)

**현재 프로덕션 벡터 검색의 표준** (Pinecone, Weaviate, Qdrant, pgvector 모두 사용)

```
Layer 2 (희소):  1 ─────── 5 ─────── 9
Layer 1 (중간):  1 ─── 3 ─ 5 ─── 7 ─ 9
Layer 0 (밀집):  1 ─ 2 ─ 3 ─ 4 ─ 5 ─ 6 ─ 7 ─ 8 ─ 9
```

- Skip List + NSW(Small World Graph) 결합
- 상위 레이어에서 대략적 탐색 → 하위 레이어로 내려가며 정밀화
- 탐색 복잡도: O(log n), 빌드 복잡도: O(n log n)

> 📖 **용어 설명**
> - **HNSW(Hierarchical Navigable Small World)**: 벡터를 계층적 그래프로 저장해 빠르게 유사한 벡터를 찾는 ANN 알고리즘. 2016년 발표 후 현재 가장 널리 쓰이는 벡터 인덱스.
> - **Small World Graph**: "6단계 분리 법칙"처럼 세상의 어떤 두 노드도 몇 번의 이동으로 연결될 수 있는 그래프. 지름길이 많아 탐색이 빠름.
> - **계층(Layer)**: HNSW는 여러 층(레이어)으로 구성. 상위 레이어일수록 노드 수가 적고 연결이 희소해 빠른 탐색 가능. 하위 레이어는 모든 노드가 존재해 정밀 탐색.
> - **희소(Sparse)**: 연결이 적고 듬성듬성한 상태. 상위 레이어의 노드는 몇 개의 "고속도로" 같은 연결만 갖고 있어 멀리 빠르게 이동 가능.
> - **밀집(Dense)**: 연결이 많고 촘촘한 상태. 하위 레이어에서 정확한 최근접 이웃을 세밀하게 탐색.
> - **O(log n)**: 데이터가 2배 늘어도 탐색 시간이 1 증가하는 로그 복잡도. 1억 개의 벡터도 약 27번의 이동으로 탐색 가능. KNN의 O(n)과 비교하면 압도적으로 빠름.
> - **Pinecone, Weaviate, Qdrant, pgvector**: HNSW를 사용하는 대표적인 벡터 데이터베이스/라이브러리.

### IVF (Inverted File Index)

FAISS의 대표적 방식.
1. 전체 벡터를 K개 클러스터로 군집화 (K-Means)
2. 쿼리 → 가장 가까운 클러스터 N개 탐색
3. 해당 클러스터 내에서만 KNN

> 📖 **용어 설명**
> - **IVF(Inverted File Index)**: 벡터를 여러 클러스터(그룹)로 나눠 저장하고, 검색 시 전체가 아닌 가까운 클러스터만 탐색하는 ANN 방법. "미국 도시 검색 시 전 세계를 다 뒤지지 않고 북미 그룹만 검색"하는 것과 유사.
> - **FAISS(Facebook AI Similarity Search)**: Meta(Facebook)가 오픈소스로 공개한 고성능 벡터 검색 라이브러리. IVF, HNSW, PQ 등 다양한 ANN 알고리즘을 제공.
> - **K-Means(K-평균 군집화)**: 데이터를 K개의 그룹으로 나누는 클러스터링 알고리즘. 각 그룹의 중심(centroid)을 계산하고, 각 데이터를 가장 가까운 중심의 그룹에 배정. 반복적으로 중심을 갱신해 수렴.
> - **클러스터(Cluster)**: 비슷한 특성을 가진 데이터끼리 모인 그룹. 비슷한 주제의 문서들이 같은 클러스터에 배치됨.

### 벡터 압축: Product Quantization (PQ)

- 고차원 벡터를 서브벡터로 분할 → 각각 코드북으로 양자화
- 768차원 float32 (3072 bytes) → 96 bytes로 32x 압축
- 정확도 약간 손실, 메모리/속도 대폭 개선

> 📖 **용어 설명**
> - **Product Quantization(PQ, 곱 양자화)**: 고차원 벡터를 여러 서브벡터로 나눠 각각 압축하는 기법. "고화질 이미지를 JPEG로 압축"하는 것과 유사하게 약간의 품질 손실로 크기를 대폭 줄임.
> - **서브벡터(Sub-vector)**: PQ에서 원본 벡터를 나눈 작은 조각. 768차원 벡터를 8조각으로 나누면 각 서브벡터는 96차원.
> - **코드북(Codebook)**: 서브벡터 공간의 대표값(centroid) 목록. 예를 들어 256개의 대표값을 정해두고, 각 서브벡터를 가장 가까운 대표값의 번호(1바이트)로 대체. 96차원 float32(384바이트) → 1바이트로 압축.
> - **양자화(Quantization)**: 연속적인 값을 이산적인(불연속적인) 값으로 근사하는 과정. MP3로 음악을 압축하거나 JPEG로 이미지를 압축하는 것도 양자화의 일종.
> - **float32**: 32비트 부동소수점 숫자. 4바이트. 벡터의 각 차원이 기본적으로 float32로 저장됨. 768차원이면 768 × 4 = 3072 바이트.

## 7. 하이브리드 검색 파이프라인

### 현대 검색 시스템의 표준 구조

```
쿼리
  │
  ├─[Lexical]─→  BM25 / Inverted Index  ─→ 후보 집합 A
  │
  └─[Semantic]─→ Dense Retrieval (ANN)  ─→ 후보 집합 B
                                              │
                              RRF / Score Fusion (두 결과 병합)
                                              │
                                        Re-Ranker (Cross-Encoder)
                                              │
                                         최종 결과 Top-K
```

> 📖 **용어 설명**
> - **Lexical 검색(어휘 검색)**: BM25처럼 단어가 실제로 문서에 있는지 글자 단위로 확인하는 검색. 정확한 키워드, 코드, 고유명사 검색에 강함.
> - **Semantic 검색(의미론적 검색)**: 임베딩 벡터의 유사도로 의미가 비슷한 문서를 찾는 검색. 동의어, 유사 표현 검색에 강함.
> - **Dense Retrieval(밀집 검색)**: 고차원 임베딩 벡터를 사용하는 검색 방식. 모든 차원에 숫자가 채워져 있어 "밀집(Dense)"이라고 부름. Sparse(희소) 방식인 BM25와 대비.
> - **후보 집합(Candidate Set)**: 최종 결과가 될 가능성이 있는 문서들의 집합. 전체에서 수천~수만 개로 추린 후보.
> - **Score Fusion(스코어 결합)**: 서로 다른 시스템에서 나온 점수를 하나의 점수로 합치는 과정. BM25 점수(0~10)와 코사인 유사도(0~1)처럼 스케일이 달라 단순 합산 불가.
> - **Top-K**: 점수 상위 K개의 결과만 반환. K=10이면 1~10위 결과만.

### RRF (Reciprocal Rank Fusion)

두 랭킹 결과를 점수 없이 순위만으로 병합.

```
RRF(d) = Σ 1 / (k + rank_i(d))    (k=60이 일반적)
```

- 스코어 스케일이 다른 두 시스템을 안전하게 병합 가능
- Elasticsearch, Vespa에 내장

> 📖 **용어 설명**
> - **RRF(Reciprocal Rank Fusion, 역순위 결합)**: 여러 랭킹 목록의 결과를 점수 없이 순위만으로 병합하는 방법. 1위 문서는 1/61, 2위는 1/62, ... 처럼 순위가 높을수록 더 큰 가중치. 점수 스케일이 달라도 안전하게 병합 가능.
> - **k=60**: RRF 공식의 상수. 60은 경험적으로 좋은 성능을 내는 값으로 알려짐. 상위권 문서들 사이의 점수 차이를 완화해 급격한 변동을 방지.
> - **Vespa**: 야후가 만든 오픈소스 검색 플랫폼. Elasticsearch와 함께 대규모 검색 시스템에서 많이 사용.

### Re-Ranking (Cross-Encoder)

- **Bi-Encoder** (임베딩): 쿼리/문서 별도 인코딩 → 빠름, ANN 가능
- **Cross-Encoder**: 쿼리+문서 쌍을 함께 입력 → 느리지만 정확도 최상
- 실제 운용: Bi-Encoder로 100개 후보 → Cross-Encoder로 Top-10 재랭킹

> 📖 **용어 설명**
> - **Re-Ranking(재랭킹)**: 1차 검색으로 추린 후보 목록을 더 정밀한 모델로 다시 정렬하는 과정. 빠른 1차 검색 + 정확한 2차 재랭킹의 2단계 구조.
> - **Bi-Encoder(바이 인코더)**: 쿼리와 문서를 **별도로** 인코딩해 각각의 임베딩 벡터를 만든 후 코사인 유사도로 비교. 문서 임베딩은 미리 계산해 저장 가능 → ANN으로 빠른 검색 가능. 정확도는 Cross-Encoder보다 낮음.
> - **Cross-Encoder(크로스 인코더)**: 쿼리와 문서를 **하나의 입력**으로 합쳐 AI 모델에 넣고 관련도 점수를 직접 출력. 쿼리-문서 간 세밀한 상호작용을 포착해 정확도 최상. 단, 모든 문서 쌍을 하나씩 처리해야 해서 수백만 문서에 직접 적용 불가 → 후보 100개로 줄인 후 사용.

## 8. 최신 트렌드: 뉴럴 랭킹 & RAG

### Dense Retrieval (밀집 검색)

- **DPR (Dense Passage Retrieval)**: 쿼리 인코더 + 문서 인코더를 대조 학습
- **Sentence-BERT**: 문장 수준 의미 임베딩
- **ColBERT**: Late Interaction — 토큰 단위 유사도 행렬로 세밀한 매칭

> 📖 **용어 설명**
> - **Dense Retrieval(밀집 검색)**: 텍스트를 고차원 벡터(임베딩)로 변환한 후 벡터 유사도로 검색하는 방식. "단어가 일치하는가"가 아닌 "의미가 유사한가"를 기준으로 검색.
> - **DPR(Dense Passage Retrieval)**: Meta AI가 2020년 발표한 오픈 도메인 질문 응답을 위한 Dense Retrieval 모델. 쿼리용 BERT 인코더와 문서용 BERT 인코더를 **대조 학습**으로 훈련.
> - **대조 학습(Contrastive Learning)**: "정답 쌍은 가깝게, 오답 쌍은 멀게" 임베딩하도록 학습하는 방법. "자동차 수리" 쿼리와 "차량 정비 방법" 문서는 가깝게, "고양이 키우기" 문서는 멀게 학습.
> - **Sentence-BERT**: BERT를 문장 수준 유사도에 최적화한 모델. BERT는 원래 두 문장을 한 번에 입력해야 유사도를 계산할 수 있었지만, Sentence-BERT는 각 문장을 독립적으로 임베딩 → 미리 문서 임베딩 저장 가능.
> - **ColBERT(Contextualized Late Interaction over BERT)**: Bi-Encoder처럼 빠르고 Cross-Encoder처럼 정확한 중간 방식. 쿼리와 문서를 각각 토큰 단위로 임베딩 후, 모든 토큰 쌍의 유사도 행렬을 계산해 세밀한 매칭.
> - **Late Interaction(레이트 인터랙션)**: 쿼리와 문서를 각각 인코딩한 후(Late = 나중에) 세밀한 상호작용을 계산하는 방식. Bi-Encoder보다 정확하고 Cross-Encoder보다 빠름.
> - **토큰(Token)**: AI 모델이 처리하는 텍스트의 최소 단위. 단어보다 더 세밀하게 나뉨. "unhappiness" → ["un", "happiness"] 또는 ["un", "happ", "iness"]. BERT는 보통 서브워드 단위로 토크나이징.

### RAG (Retrieval-Augmented Generation)

```
[쿼리]
  → Retriever (BM25 + Dense Hybrid)
  → 관련 문서 Top-K 추출
  → LLM에 컨텍스트로 주입
  → 답변 생성
```

- 검색 정확도(Precision)가 LLM 답변 품질에 직결
- **Chunking 전략**: 문서를 어떻게 나누느냐가 성능 좌우
- **Re-ranking**: LLM 비용 절감을 위해 관련 문서만 추려서 전달

> 📖 **용어 설명**
> - **RAG(Retrieval-Augmented Generation, 검색 증강 생성)**: LLM이 학습 데이터만으로 답변하는 것을 넘어, 실시간으로 관련 문서를 검색해 그 내용을 참고하며 답변을 생성하는 방식. 최신 정보 반영 및 환각(Hallucination) 감소에 효과적.
> - **Retriever(리트리버)**: RAG에서 쿼리와 관련된 문서를 검색하는 컴포넌트. BM25, Dense Retrieval, Hybrid 등 다양한 방식 사용.
> - **컨텍스트(Context)**: LLM에 답변 생성을 위해 제공하는 배경 정보. 검색된 관련 문서들이 컨텍스트로 LLM의 프롬프트에 포함됨.
> - **주입(Injection)**: 외부 정보를 LLM의 입력(프롬프트)에 넣어주는 것.
> - **Precision(정밀도)**: 검색된 결과 중 실제로 관련 있는 문서의 비율. Precision = (실제 관련 문서 수) / (검색된 전체 문서 수). RAG에서 Precision이 낮으면 관련 없는 내용이 LLM에 들어가 잘못된 답변 생성.
> - **Chunking(청킹)**: 긴 문서를 검색에 적합한 작은 단위로 나누는 과정. 너무 작으면 문맥 손실, 너무 크면 관련 없는 내용 포함. 적절한 크기와 겹침(overlap) 설정이 RAG 성능의 핵심.
> - **환각(Hallucination)**: LLM이 사실이 아닌 정보를 자신 있게 출력하는 문제. RAG는 검색된 사실 기반 문서를 제공해 환각을 줄임.

### Sparse + Dense 하이브리드의 우위

| 상황 | 우수한 방식 |
|------|------------|
| 키워드, 코드, 고유명사 | Sparse (BM25) |
| 의미적 유사, 동의어 | Dense (Embedding) |
| 일반 프로덕션 | **Hybrid** (항상 Hybrid가 단일보다 우수) |

> 📖 **용어 설명**
> - **Sparse(희소)**: 대부분의 값이 0인 벡터. BM25 기반 검색에서 "단어 존재 여부"를 표현할 때 전체 어휘(수십만 개) 중 실제로 등장한 단어만 값이 있고 나머지는 0 → 대부분이 0이므로 "희소".
> - **Dense(밀집)**: 모든 차원에 0이 아닌 값이 채워진 벡터. 임베딩 벡터는 768개 차원 모두에 의미 있는 숫자가 채워짐.
> - **프로덕션(Production)**: 실제 사용자가 사용하는 라이브 서비스 환경. 개발(Development)이나 테스트(Test) 환경과 반대.

### 현재 최전선 (2025)

- **ColPali**: 이미지/PDF 페이지를 통째로 임베딩 (OCR 없이 멀티모달 검색)
- **Matryoshka Embeddings**: 임베딩을 잘라도 유효 → 저장 비용 vs 정확도 트레이드오프
- **Learned Sparse Retrieval (SPLADE)**: Dense의 의미 이해 + Sparse의 해석 가능성 결합

> 📖 **용어 설명**
> - **ColPali**: 2024년 발표된 멀티모달 문서 검색 모델. PDF를 텍스트로 변환(OCR)하지 않고 페이지 이미지를 통째로 임베딩해 검색. 표, 차트, 이미지가 포함된 문서 검색에 강력.
> - **OCR(Optical Character Recognition, 광학 문자 인식)**: 이미지 속 텍스트를 컴퓨터가 읽을 수 있는 텍스트로 변환하는 기술. 스캔 문서, PDF에서 텍스트를 추출할 때 사용. ColPali는 OCR 없이 이미지 자체를 이해.
> - **Matryoshka Embeddings(마트로시카 임베딩)**: 러시아 인형(마트로시카)처럼 큰 임베딩 안에 작은 임베딩이 들어있는 구조. 1536차원 임베딩의 앞 256차원만 잘라도 유효한 임베딩으로 동작. 저장 비용과 검색 속도를 유연하게 조절 가능.
> - **트레이드오프(Trade-off)**: 한 쪽을 얻으면 다른 쪽을 잃는 상충 관계. 임베딩을 줄이면 속도/비용은 좋아지지만 정확도는 낮아짐.
> - **SPLADE(Sparse Lexical and Expansion)**: Dense 모델의 의미 이해 능력과 Sparse 방식의 해석 가능성(어떤 단어가 왜 매칭됐는지 설명 가능)을 결합한 Learned Sparse Retrieval 모델. BM25처럼 희소 벡터를 쓰지만 AI가 단어의 의미를 확장해 어휘 불일치 문제도 해결.
> - **해석 가능성(Interpretability)**: AI 모델의 판단 근거를 사람이 이해할 수 있는 정도. Dense 임베딩은 숫자 덩어리라 왜 유사하다고 판단했는지 알기 어렵지만, Sparse 방식은 어떤 단어가 매칭됐는지 명확히 알 수 있음.
