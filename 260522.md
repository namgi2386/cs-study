## [velog] Vite의 개발 서버는 왜 빠를까: ESM, pre-bundling, production build 이해하기

### 빌드 시스템

- 빌드 : 소스코드를 컴파일+번들링하여 실행가능한 형태로 변환하는 것
  - 컴파일 : 같은 추상화 수준의 언어끼리 변환 (TS → JS, ES6 → ES5)
    - 컴파일러 : 고수준 언어를 저수준 언어로 변환
    - 트랜스파일러 : 사실상 FE는 트랜스파일만 하는 느낌
  - 번들링 : 번들러(webpack, vite 등)가 모듈을 하나의 파일로 묶음
    - 모듈 : 기능단위의 코드 파일 (import/export로 재사용하는 그런것들)
    - 번들링 최적화 방식
      - 코드 최소화 : 공백, 주석제거, 변수명 단축 등
      - 트리 쉐이킹 : 사용하지 않는 코드제거
      - Code splitting: 파일을 여러 청크로 나눔
      - Lazy loading: 필요한 시점에만 모듈 로드
      - Asset optimization: 이미지 압축, 포맷 변환 등
      - Source map 생성: 디버깅용
  - 실행 가능한 형태란?
    - FE입장에서는 : 최적화된 JS, CSS, HTML, 정적 파일
- 빌드 도구 (번들러)
  - Webpack : Babel트랜스파일러 사용. 레거시 방식
  - Vite (래퍼)(번들러아님)
    - 기본 래퍼: esbuild(개발) + Rollup(프로덕션) 두가지 제공
      - esbuild : 자체 내장 트랜스파일러 사용. Go기반, 매우빠름, 최적화 능력은 부족함 => 그래서 개발시 사용
      - rollup : 자체 트랜스파일러 없어서 플러그인 필요함. JS기반, 느림, 고급 최적화 강력함 => 그래서 프로덕션에 사용함
    - 확장 래퍼: Babel, SWC 등 선택적 플러그인 지원
  - Turbopack : SWC트랜스파일러 사용. (next.js 의 선택사항)
    - 참고로, Next.js 에서는 Next.js 자체 번들링 로직으로 번들링함. (추후 Turbopack통합 예정)

### Vite

- vite는 래퍼이다. 번들러아니다. 빌드 최적화 도구일 뿐이다.
  - esbuild(개발) + Rollup(프로덕션) 두가지 번들러를 제공하며, Babel, SWC 등 선택적 플러그인을 지원한다.
  - 자체 로직: pre-bundling, HMR(Hot Module Replacement), 캐싱, 최적화 등 지원함
  - Vite는 개발 모드에서 source code를 미리 번들링하지 않지만, dependencies는 pre-bundling하고 production build에서는 다시 번들링한다.
  - 개발모드시, 소스코드를 그대로 브라우저에 제공하며, 브라우저가 import를 만났을때, 서버에 요청하며, 서버는 필요한 모듈만 컴파일하는 on-demand 방식이다.
- Source Code에는 native ESM 기반 on-demand transform 이 적용된다.
  - On-Demand Transform : 요청이 들어올때마다, 그파일만 변환한다. (빌드다시안함)
    - 이게 가능한 이유는 브라우저가 ESM을 직접 로드할 수 있으며, 필요할때마다 서버에 요청할 수 있기 때문이다.
  - type checking : 전체 의존성을 분석해야만 하는 type checking은 On-Demand Transform 과정에서 불가능하다. 때문에 사용자는 tsc --noEmit로 따로 type checking 해야만 한다.
    - 참고로 Next.js와 Expo의 Metro bundler에서는 백그라운드에서 자동으로 type checking 수행함.
- pre-bundling
  - npm install 하면 의존성 설치하며 node_modules 생성됨
  - 최초 npm run dev (cold start)하면, pre-bundling 한다.
    - COMMONJS/UMD 를 ESM 모듈시스템으로 변환하여 node_modules/.vite 에 캐싱한다.
  - 브라우저가 모듈 요청하면 캐싱된 모듈 전달한다. 없으면 즉각 해당 모듈만 변환해서 전달한다.
- production
  - production에 unbundled ESM을 그대로 배포하면 nested import 때문에 추가 network round trip이 생긴다.
  - 작은 파일을 압축 캐싱하는 효율보다 큰 번들을 압축하는 것이 훨씬 유리하다.
  - 단, 모든 파일을 하나로 합치지는 않는다. Code Splitting로 최적화한다.
  - 동적으로 필요한 코드(Dynamic Import)는 async chunk로 분리될 수 있다.
    - Vite는 빌드 시점에 모듈 그래프를 미리 분석한다. (의존성 해석)
    - production build는 단순히 파일을 합치는 작업이 아닌 브라우저가 실제로 코드를 로드하는 과정에서 생길 수 있는 round trip까지 줄이는 방향으로 결과물을 만든다.
    - 특히 barrel export 패턴을 사용하면 의존성 구조가 명확해져서 번들 최적화 판단에 도움이 됩니다.
- Rolldown, Oxc
  - Rolldown: 기존의 esbuild와 Rollup의 통합인 Rolldown을 개발중이다.
    - Rust라서 빠르며, esbuild/Rollup 차이로 인한 오류를 방지하기 위함이다.
  - Oxc: 기존의 JS기반 도구들인 Babel(transpiler), ESLint(linter), Prettier(formatter)를 Rust기반 통합 도구로 만든 것.

## [velog] Vite + React 앱 번들 사이즈 86% 줄이기 — manualChunks와 React.lazy 실전 적용기

### 웹 통신 프로토콜의 버전

- HTTP/1.1과 HTTP/2로 구분됨
  - 99% 사용자는 HTTP/2 지원 브라우저 사용중이다. 때문에 HTTP/1.1 무시해도됨
  - HTTP/1.1는 하나씩요청하며 HTTP/2는 병렬요청이 된다.

### 번들 청크 파일 분석

- 분석 방법
  - 방법1 청크파일 분석 라이브러리 : Vite Plugin - Visualizer 사용
    - 각 청크에 어떤 모듈이 얼마나 포함되었는지 시각화됨
  - 방법2 Source Map 확인
    - 빌드시 .js.map 파일이 생성되면 `npm run build -- --sourcemap` 이후 개발자도구 sources 탭에서 확인가능
  - 방법3 번들 분석 도구 사용
    - `npm install --save-dev webpack-bundle-analyzer`
    - `npm install --save-dev source-map-explorer`
- Vendor 해시 : 번들링 후 index-Cx4FtwR9.js 생성될때의 해시값
  - 해시값 변경 추적을 통해 on-demand 가능함
  - 단, 이때문에 여러 모듈이 하나의 chunk로 번들링 되었을때, 코드 한줄만 바뀌어도 해시값이 새로 할당되며, 새로 번들링 해야하는 문제가 발생한다.
  - chuck로 나누는 것은 code spliting 에 의해 나뉘는 것이며, dev환경에서는 source code가 번들링되지 않는다. 때문에, production 환경에서만 논의하면 된다.
  - 물론, vite에서는 npm 라이브러리와 source code를 구분해서 번들링한다.
  - 그럼에도 수동 manualChunks 가 필요하다.

### 번들 최적화 기법

#### manualChunks

- 청크를 묶는 기준을 설정할 수 있다.
  - vite.config.ts 에서 수정가능
  - vendor-react, vendor-query, ui모듈, app모듈 등으로 구분 하는 방식
- 어떤 기준으로 묶을까
  - 변경 빈도와 크기를 기준으로 나눠라.
  - 변경 빈도가 낮은 라이브러리 React, lodash를 vendor 청크로 분리
  - 재사용되지만, 빈도가 낮은 라이브러리 components 청크로 분리
  - 유틸/컴포넌트/페이지 로 분리하거나
  - 라이브러리별로 분리하거나
  - 폴더(페이지)별로 분리하거나
