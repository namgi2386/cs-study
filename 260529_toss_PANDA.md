# 토스플레이스 데이터봇 판다(PANDA)

> 원문: toss.tech [토스플레이스 데이터봇 '판다(PANDA)'](https://toss.tech/article/da-assistant-panda)

---

## PANDA

토스플레이스 내부 데이터 분석 어시스턴트. 팀원이 자연어로 질문하면 데이터를 조회·분석해서 답변해주는 슬랙봇 형태의 도구다.

기술 구조를 뜯어보면 **Text-to-SQL RAG + Agentic Loop** 조합이다.

```t
사용자 질문
  → AI 모델 [claude Sonnet 4.6] (질문 이해, 필요한 테이블 판단)
  → RAG (Scoring & Ranking으로 관련 테이블 스키마 검색·전달)
  → AI 모델 (SQL 작성)
  → Snowflake (SQL 실행)
  → AI 모델 (결과 검증, 필요시 재시도, 최종 답변 생성)
  → 사용자
```

---

## 기술적으로 특별한 건 없다, 그런데 왜 읽을 만한가

판다의 기술 스택 자체는 업계에서 이미 알려진 패턴들이다.

- Text-to-SQL RAG
- Agentic Loop + 재시도 → ReAct 패턴
- Tool Calling → LLM 기본 기능
- LangChain, Vanna.ai 같은 오픈소스 도구들도 이미 존재

그렇다면 뭐가 특별한가.

---

## 진짜 핵심은 기술이 아니다

판다를 만들기 전에 토스가 먼저 한 일이 있다.

1. SSOT 정비
테이블 네이밍부터 통일했다. `fact_device_error_log`, `dim_merchant_category` 같은 규칙을 만들고, 컬럼명에도 `is_merchant_active`, `min_order_amount` 형태의 일관된 규칙을 적용했다. 테이블과 컬럼 Description도 전부 채웠다.

2. 비즈니스 언어 정의
"활성 매장이 뭐야?" 같은 질문에 AI가 정확히 답하려면 그 기준 자체가 정의되어 있어야 한다. 데이터 분석가들이 모여서 사람마다 다르게 쓰던 용어를 하나로 맞췄다.

이걸 먼저 안 하면 어떻게 되는지 글에서 직접 언급한다. 토큰 낭비, 매번 다른 테이블 선택, 잘못된 비즈니스 로직 적용. AI가 멍청한 게 아니라 참고할 기준 자체가 없었던 것이다.

순서가 핵심이다. AI를 먼저 붙이고 데이터를 나중에 정비하려는 시도가 왜 실패하는지 이 글이 잘 설명해준다.

---

## Scoring & Ranking 시스템은 흥미롭다

단순 벡터 유사도 검색이 아니라 데이터 신뢰도 계층을 가중치로 결합한 구조다.

```t
최종 점수 = 유사도 점수 × 계층 가중치

SSOT (전사 주요 지표)  × 4
표준 마트              × 3
도메인 마트            × 2
원시 데이터 & 로그     × 1
```

AI가 매번 다른 테이블을 고르는 "일관성" 문제를 모델 튜닝으로 해결하려 하지 않고, 테이블 선택 기준 자체를 구조화해서 해결한 접근이 개발자 입장에서 깔끔하다고 느꼈다.

---

## 왜 지금 만들었을까

그들만이 알 것 이다. 그럼에도 왜 이제서야 만들었을까 망상해보자. 수만가지 이유에 의해서 지금 만들었겠지.

- GPT-4급 LLM이 SQL을 신뢰할 수준으로 생성하기 시작한 게 2023년
- Tool Calling이 안정화된 것도 2023년 중반
- LLM API 비용이 대량 사용 가능한 수준으로 내려온 것도 최근 (다시 비싸질 것 같은데...)
- 토스 내부의 SSOT 정비가 완료된 것도 판다 직전

외부 기술 조건과 내부 데이터 인프라가 동시에 준비된 타이밍이기에 했겠지 싶다.

---

## 결국 이 글이 말하는 것

> "AI를 잘 활용한다는 것은 꼭 복잡한 기술을 적용하는 데 있지 않다. 결국 중요한 것은 비즈니스 문제를 해결하고 실제 Pain Point를 풀어내는 것이었어요."

데이터 요청의 70%가 단순 수치 추출이라는 걸 먼저 파악하고, 그것만 해결하는 도구를 만들었다. 모든 데이터 분석을 AI로 대체하려 한 게 아니라 가장 빈번한 Pain Point를 정확히 겨냥했다.

한 달 만에 만들어서 출시 일주일 내 직원 절반이 쓰고 현재 70%가 사용 중인 도구가 됐다는 결과가 그 판단이 맞았다는 증거다.

개발자로서 인상적이었던 건 기술 선택이 아니라 "문제를 먼저 정의하고 그에 맞는 최소한의 기술을 골랐다"는 태도다. AI 시대에 오히려 더 중요해진 역량인 것 같다.
