## 1. 캐시란 무엇인가

캐시는 이미 가져온 데이터를 저장해두고, 동일한 요청이 올 때 원본 소스에 다시 접근하지 않고 저장된 데이터를 재사용하는 기법이다.

**캐싱의 이점**
- 네트워크 트래픽 감소
- 사용자 경험 개선 (빠른 로딩)
- 서버 부하 감소

**캐싱의 위험**
- 변경된 데이터가 반영되지 않아 정합성 문제 발생
- 개인정보가 공유 캐시에 남아 보안 문제 발생

---

## 2. 브라우저 캐시의 종류와 저장 위치

| 캐시 종류 | 저장소 | 유지 기간 | 용도 |
|---------|-------|---------|------|
| HTTP 캐시 | 디스크 | Cache-Control max-age까지 | JS, CSS, 이미지 등 정적 자산 |
| 메모리 캐시 (RAM) | RAM | 브라우저 탭 살아있는 동안 | 같은 세션 내 빠른 재사용 |
| localStorage | 디스크 | 반영구적 (사용자 삭제 전까지) | 영구 저장 데이터 |
| sessionStorage | 메모리 | 탭 종료 시 삭제 | 세션별 임시 데이터 |
| 쿠키 | 디스크 | expires / max-age까지 | 서버에 자동 전송되는 인증 데이터 |
| React Query | RAM | gcTime 설정값까지 | API 응답 클라이언트 캐시 |

### HTTP 캐시의 물리적 위치

```
Chrome (Windows): C:\Users\{username}\AppData\Local\Google\Chrome\User Data\Default\Cache
Firefox (Windows): C:\Users\{username}\AppData\Local\Mozilla\Firefox\Profiles\{profile}\cache2
```

브라우저를 종료하거나 컴퓨터를 재부팅해도 디스크에 유지된다. max-age가 만료되거나 사용자가 수동으로 삭제해야 사라진다.

### 캐시 용량 한도

```
Chrome:  ~50MB
Firefox: ~350MB
Safari:  ~50MB
```

용량 초과 시 LRU(Least Recently Used) 방식으로 오래된 것부터 자동 삭제된다.

## 3. HTTP Cache-Control 헤더

서버가 응답 시 포함하는 HTTP 헤더로, 브라우저와 중간 프록시(CDN, nginx)에게 이 응답을 어떻게 캐시할지 지시한다.

### 주요 디렉티브

| 디렉티브 | 의미 |
|---------|------|
| `max-age=N` | N초 동안 캐시 유효. 이 시간 내에는 서버에 재요청하지 않음 |
| `no-cache` | 캐시는 저장하되, 사용 전 반드시 서버에 유효성 검증 요청 |
| `no-store` | 캐시 자체를 저장하지 않음. 매번 서버에서 새로 받음 |
| `public` | 공유 캐시(CDN, 프록시)에도 저장 허용 |
| `private` | 브라우저 개인 캐시에만 저장. 공유 캐시 불가 |
| `immutable` | max-age 내에서는 서버 검증 없이 캐시를 그대로 사용 |
| `s-maxage=N` | 공유 캐시(CDN)에만 적용되는 max-age. 브라우저는 max-age를 따름 |
| `must-revalidate` | max-age 만료 후 반드시 서버 검증. 오프라인 상태에서 만료 캐시 사용 금지 |

### no-cache vs no-store 차이

```
no-cache: 캐시 저장 O, 사용 전 서버 확인 필수 (ETag/Last-Modified 활용)
no-store: 캐시 저장 X, 항상 서버에서 새로 받음
```

`no-cache`는 "캐시하지 마라"가 아니라 "캐시하되 쓰기 전에 확인하라"는 의미다.

### 설정 주체

- **프론트엔드 JS 코드**로는 응답 헤더를 설정할 수 없다
- **서버(Next.js, Express, FastAPI, nginx)** 에서 응답 헤더로 설정한다
- 브라우저는 서버가 내려준 헤더를 읽고 자동으로 캐싱 여부를 결정한다

## 4. 정적 자산 vs 동적 데이터 캐싱 전략

### 정적 자산 — 공격적 캐싱

```
JS 번들, CSS → Cache-Control: public, max-age=31536000, immutable
```

안전한 이유: **콘텐츠 해싱(Content Hashing)**

빌드 시 파일명에 파일 내용의 해시값을 삽입한다.

```
빌드 전: main.js
빌드 후: main.abc123f.js   ← 내용이 바뀌면 해시도 바뀜
```

내용이 바뀌면 파일명(URL)이 바뀌므로 캐시 미스가 발생해 자동으로 새로 다운로드된다. 내용이 동일하면 URL도 동일하므로 1년 캐시를 써도 안전하다.

### 동적 데이터 — 캐싱 금지

```
API 응답, 개인정보 → Cache-Control: no-store, private
```

자주 바뀌거나 사용자별로 다른 데이터는 캐싱하면 정합성 문제 또는 개인정보 노출 위험이 생긴다.

### HTML — no-store (핵심)

```
HTML → Cache-Control: no-store
```

HTML은 JS/CSS의 "목차" 역할을 한다.

```html
<!-- 배포 v1 HTML -->
<script src="/static/chunks/abc111.js">

<!-- 배포 v2 HTML (배포 후 새 해시) -->
<script src="/static/chunks/abc999.js">
```

HTML이 캐싱되면 새 배포 후에도 구버전 JS URL을 참조하게 된다. HTML은 반드시 매번 최신 버전을 받아야 하므로 no-store로 설정한다.

**구조적 의도:**
```
HTML    → no-store  → 항상 최신 목차 보장
JS/CSS  → 1년 캐싱  → HTML이 올바른 URL을 가리키므로 안전
```

## 5. Next.js의 자동 캐시 처리

Next.js는 별도 설정 없이도 아래를 자동으로 처리한다.

```
/_next/static/**  (JS, CSS 번들)  → Cache-Control: public, max-age=31536000, immutable
HTML 페이지                        → Cache-Control: no-store
```

이는 Next.js만의 특징이 아니라 Vite, Nuxt 등 **현대 웹 프레임워크의 표준 패턴**이다. 순수 Express나 직접 서버를 구성하면 이 모든 것을 수동으로 설정해야 한다.

### Next.js Data Cache (서버 사이드)

Next.js App Router에는 별도의 서버 사이드 fetch 캐시가 있다.

```typescript
// Next.js 서버에서 백엔드 API를 호출할 때
fetch(url, { cache: 'no-store' })   // 서버 캐시 비활성화
fetch(url, { cache: 'force-cache' }) // 서버 캐시 활성화
```

이것은 **브라우저 캐시와 전혀 다른 레이어**다. Next.js 서버가 백엔드 API 응답을 서버 메모리에 캐싱할지 여부를 제어한다.

### Cache-Control 직접 설정이 필요한 경우

Next.js가 자동으로 처리하지 못하는 영역:

1. **`/public/` 폴더 파일** — 해시가 없는 고정 URL이므로 직접 설정 필요
2. **CDN 사용 시** — `s-maxage`로 CDN 캐시 제어
3. **API Route에서 특정 응답 캐싱** — 수동 설정 필요

## 6. 아키텍처별 캐시 책임 분담

### Next.js + FastAPI 구조

```
브라우저
  ↓ (HTTP 요청)
Next.js 서버  ← 브라우저는 Next.js 응답 헤더만 본다
  ↓ (서버-서버 통신)
FastAPI
```

브라우저는 Next.js가 내려주는 헤더만 본다. FastAPI가 Cache-Control 헤더를 설정해도 브라우저에 직접 전달되지 않는다 (Next.js가 가로채기 때문).

### 책임 분담

| 계층 | Cache-Control 책임 | 이유 |
|------|------------------|------|
| **Next.js** | 정적 자산, HTML → 자동 처리 | 브라우저와 직접 통신하는 주체 |
| **FastAPI** | SSE 스트림 헤더 | Next.js를 거치지 않는 직접 스트리밍 |
| **nginx/CDN** | 프록시 캐시, s-maxage | 엣지 캐싱 필요 시 |

### 프레임워크별 신경 써야 하는 정도

| 환경 | Cache-Control 신경 정도 |
|------|----------------------|
| 순수 Node.js/Express | 높음 — 모든 설정 수동 |
| Next.js | 낮음 — 자동 처리 |
| FastAPI (독립 서버) | 보통 — API 응답별 직접 설정 |
| FastAPI (Next.js 뒤) | 낮음 — Next.js가 커버 |

## 7. React Query와 HTTP 캐시의 차이

가장 많이 혼동하는 개념이다.

| 구분 | HTTP 캐시 (브라우저 디스크) | React Query 캐시 |
|------|--------------------------|----------------|
| 저장 위치 | 디스크 | JS 메모리 (RAM) |
| 유지 기간 | max-age 만료까지 | gcTime 만료까지 (탭 닫으면 삭제) |
| 제어 방법 | Cache-Control 헤더 | staleTime, gcTime 옵션 |
| 대상 | JS, CSS, 이미지 파일 | API 응답 데이터 |
| 탭 닫으면 | 유지 | 삭제 |

React Query는 HTTP Cache-Control과 무관하다. 브라우저 네트워크 캐시 위에서 독립적으로 동작하는 **애플리케이션 레이어 캐시**다.

### React Query 주요 옵션

```typescript
useQuery({
  staleTime: 60 * 1000,   // 이 시간 동안 fresh로 간주, 서버 재요청 안 함
  gcTime: 5 * 60 * 1000,  // 컴포넌트 언마운트 후 이 시간 동안 캐시 보관
  refetchOnWindowFocus: true,  // 탭 포커스 시 재요청 여부
})
```

## 8. nginx 캐싱 vs 브라우저 캐싱

완전히 다른 목적과 위치를 가진다.

| 구분 | 브라우저 캐싱 (Cache-Control) | nginx 캐싱 |
|------|---------------------------|-----------|
| 저장 위치 | 사용자 PC 디스크 | 서버 (nginx) |
| 목적 | 개별 사용자의 재방문 속도 개선 | 백엔드 서버 부하 감소 |
| 영향 범위 | 해당 사용자만 | 모든 사용자 |
| Cold start 효과 | 없음 (캐시가 없으므로) | 있음 (서버 응답 빠름) |
| Warm start 효과 | 매우 큼 | 있음 |

**두 캐싱은 독립적으로 작동하며 서로 영향을 주지 않는다.** 둘 다 적용하면 최적의 성능을 낼 수 있다.

## 9. 캐시와 보안

### 개인정보와 캐시

개인정보가 화면에 표시된다고 해서 디스크에 캐싱되지 않는다.

```
API 응답 (개인정보) → React Query (RAM) → 화면 렌더링
```

개인정보는 JS 메모리에만 존재하며, 탭을 닫으면 사라진다. 브라우저 디스크에 저장되는 것은 JS/CSS 코드 파일뿐이고, 이는 모든 사용자에게 동일한 코드다.

### 공용 PC에서의 위험 시나리오

```
Cache-Control: public, max-age=3600  ← API 응답에 이렇게 설정하면
→ A 사용자의 개인정보가 디스크 캐시에 저장
→ B 사용자가 같은 URL 요청 시 A의 데이터가 반환될 수 있음
```

개인 데이터 API에는 반드시 `private, no-store`를 사용해야 한다.

### 인증 쿠키 보안 설정

```python
response.set_cookie(
    httponly=True,   # JS에서 document.cookie로 접근 불가 → XSS 방지
    secure=True,     # HTTPS 연결에서만 전송 → 중간자 공격 방지
    samesite='lax'   # 크로스사이트 요청 시 쿠키 제한 → CSRF 방지
)
```

### localStorage와 보안

Zustand `persist` 미들웨어 등으로 localStorage에 데이터를 저장할 때 주의할 점:

- localStorage는 JS로 자유롭게 접근 가능 → XSS 취약점 있으면 탈취 가능
- **인증 토큰을 localStorage에 저장하면 안 된다** (HttpOnly 쿠키 사용)
- UI 설정값(필터, 레이아웃 등) 정도만 저장해야 한다
- 로그아웃 시 반드시 `localStorage.clear()` 처리

## 10. 번들 최적화와 캐시의 관계

### Cold Start vs Warm Start

| 상황 | 설명 | 캐시 효과 |
|------|------|---------|
| Cold Start | 첫 방문, 캐시 없음 | 없음. 모든 파일 다운로드 |
| Warm Start | 재방문, 캐시 있음 | 매우 큼. JS/CSS 재다운로드 없음 |
| 배포 후 재방문 | 해시 바뀐 파일만 다운로드 | 변경된 청크만 새로 받음 |

성능 최적화는 Cold Start 사용자를 기준으로 측정해야 의미 있다. DevTools Network 탭에서 "Disable cache" 체크 후 측정한다.

### React.lazy를 통한 코드 스플리팅

```typescript
// 모든 컴포넌트를 한 번에 받는 경우
import HeavyComponent from './HeavyComponent';
// → 초기 번들에 포함 → 첫 로딩 느림

// 동적 import로 필요할 때만 받는 경우
const HeavyComponent = React.lazy(() => import('./HeavyComponent'));
// → 해당 컴포넌트가 실제 렌더링될 때 별도 청크 다운로드
```

코드 스플리팅을 하면 초기 번들 크기가 줄어 Cold Start 속도가 개선된다. 분리된 청크는 Next.js가 별도 해시 파일로 빌드하여 독립적으로 캐싱된다.

## 11. 실무 경험: ChuChuTree 프로젝트 캐시 분석

### 프로젝트 스택

```
브라우저 → Next.js (App Router) → FastAPI → PostgreSQL
```

### 캐시 레이어 전체 구조

```
[디스크 캐시]
  /_next/static/**  → Cache-Control: public, max-age=31536000, immutable (Next.js 자동)
  HTML              → Cache-Control: no-store (Next.js 자동)

[서버 사이드 Data Cache]
  server.ts fetch   → cache: 'no-store' (사용자별 개인 데이터이므로 서버 캐시 비활성화)

[메모리 캐시 - React Query]
  기본 설정         → staleTime: 60분, gcTime: 60분
  공지사항          → staleTime: 0 (실시간 반영 필요)
  문제 검색         → staleTime: Infinity (세션 내 반복 타이핑 방지)

[SSE 스트림]
  Cache-Control: no-cache, no-transform
  X-Accel-Buffering: no  (nginx 버퍼링 방지)

[쿠키]
  인증 토큰         → HttpOnly, Secure(prod), SameSite=lax
  사이드바 상태      → max-age=7일
```

### 설계 결정 포인트

**1. server.ts의 `cache: 'no-store'`**

Next.js App Router는 기본적으로 서버 fetch 결과를 캐싱한다. 이 프로젝트는 로그인한 사용자별 데이터를 서버 컴포넌트에서 fetch하므로, 서버 캐시가 활성화되면 A 사용자 데이터가 B 사용자에게 반환될 수 있다. 따라서 기본값을 `no-store`로 설정하고, 캐싱이 필요한 API에서만 override하는 구조다.

**2. React Query staleTime 60분**

대부분의 데이터(캘린더, 태그 대시보드 등)가 사용자 세션 중에 자주 바뀌지 않으므로 60분 stale 정책으로 불필요한 네트워크 요청을 줄였다. 실시간성이 필요한 공지사항은 `staleTime: 0`으로 예외 처리했다.

**3. Zustand persist의 partialize**

localStorage에 영구 저장되는 Zustand 스토어들은 `partialize`로 저장 대상을 명시적으로 제한했다. 실제 문제 데이터(`problems`)는 제외하고 필터 설정값(레벨, 태그 선택 등 UI 상태)만 저장한다. 로그아웃 시 `localStorage.clear()`로 전체 초기화한다.

**4. SSE의 이중 no-cache 설정**

SSE 스트림은 Next.js Route Handler와 FastAPI 양쪽에서 모두 `Cache-Control: no-cache`를 설정했다. Route Handler에서만 설정해도 브라우저에는 충분하지만, FastAPI에서도 설정하는 이유는 nginx 등 중간 프록시가 스트림을 버퍼링하지 않도록 하기 위함이다.

### 보안 분석 결과

| 위협 | 상태 | 근거 |
|------|------|------|
| A→B 개인정보 유출 (디스크) | 안전 | 개인 데이터는 React Query(RAM)에만 존재 |
| A→B 개인정보 유출 (localStorage) | 안전 | partialize로 개인 데이터 제외, 로그아웃 시 clear |
| XSS에 의한 토큰 탈취 | 안전 | 인증 토큰은 HttpOnly 쿠키로만 관리 |
| 실시간 동기화 지연 | 주의 | staleTime 60분 + refetchOnWindowFocus: false |

**유일한 주의 사항:** staleTime 60분으로 인해 서버 데이터 변경이 최대 60분 후에 반영된다. 데이터 변경 빈도가 높아지면 해당 쿼리의 staleTime을 줄이거나 mutation 후 `invalidateQueries`를 트리거하는 방식으로 대응해야 한다.

## 요약 — 면접 대비 핵심

**Q. Cache-Control이 뭔가요?**
> HTTP 응답 헤더로, 브라우저와 중간 프록시에게 응답을 얼마나, 어떻게 캐싱할지 지시합니다. `max-age`로 유효 기간을, `private/public`으로 공유 범위를, `no-store`로 캐싱 자체를 금지할 수 있습니다.

**Q. Next.js에서 Cache-Control을 직접 설정해야 하나요?**
> 대부분은 불필요합니다. Next.js가 JS/CSS 번들에 1년 캐시를, HTML에 no-store를 자동으로 적용합니다. `/public/` 폴더 파일이나 CDN 사용 시에만 직접 설정합니다.

**Q. React Query 캐시와 HTTP 캐시의 차이는?**
> HTTP 캐시는 브라우저 디스크에 파일을 저장하고 Cache-Control 헤더로 제어합니다. React Query 캐시는 JS 메모리(RAM)에 API 응답 데이터를 저장하고 staleTime/gcTime으로 제어합니다. 탭을 닫으면 React Query 캐시는 사라지지만 HTTP 캐시는 유지됩니다.

**Q. 개인정보가 화면에 표시될 때 캐시 보안은 괜찮은가요?**
> 개인정보 데이터는 React Query(메모리)에만 존재하고 디스크에 저장되지 않습니다. 디스크에 캐싱되는 JS/CSS는 모든 사용자에게 동일한 코드 파일이므로 개인정보가 포함되지 않습니다. 다만 FastAPI 같은 독립 API 서버에서 개인정보 API 응답에 `Cache-Control: public`을 설정하면 공유 캐시에 노출될 수 있어 반드시 `private, no-store`를 사용해야 합니다.

**Q. HTML은 왜 캐싱하지 않나요?**
> HTML은 JS/CSS 파일의 URL(해시 포함)을 담고 있는 목차 역할입니다. HTML이 캐싱되면 새 배포 후에도 구버전 JS/CSS URL을 참조하게 됩니다. HTML이 no-store로 항상 최신 상태를 유지하기 때문에, JS/CSS는 1년 캐싱을 해도 안전합니다. 이 둘은 짝을 이루는 설계입니다.

> 1분 기준 약 250~300자 (보통 말하기 속도 기준)

### Version 1. 개념 중심형 (이론을 탄탄하게 보여주고 싶을 때)

캐시는 동일한 데이터를 반복해서 가져오는 비용을 줄이기 위해 결과를 임시 저장해두는 기법입니다.

웹에서 캐시는 크게 세 레이어로 나뉩니다. 첫째는 브라우저 HTTP 캐시로, 서버가 응답에 `Cache-Control` 헤더를 통해 JS나 CSS 같은 정적 파일을 디스크에 캐싱하도록 지시합니다. 둘째는 서버 측 캐시로, nginx나 CDN이 백엔드 응답을 저장해 서버 부하를 줄입니다. 셋째는 애플리케이션 레이어 캐시로, React Query 같은 도구가 API 응답을 메모리에 임시 저장합니다.

중요한 설계 원칙은 '무엇을 캐싱하느냐'입니다. JS나 CSS처럼 내용이 바뀌면 파일명(해시)이 바뀌는 정적 자산은 1년씩 공격적으로 캐싱해도 안전합니다. 반면 사용자 개인정보나 실시간 데이터는 `no-store`로 캐싱을 막아야 정합성과 보안을 지킬 수 있습니다.

---

### Version 2. 경험 중심형 (프로젝트 경험을 자연스럽게 연결할 때)

캐시는 자주 쓰는 데이터를 가까운 곳에 저장해 응답 속도를 높이고 서버 부하를 줄이는 기법입니다.

실제로 Next.js와 FastAPI를 사용한 프로젝트에서 캐시 레이어를 직접 분석하고 설계한 경험이 있습니다. Next.js는 JS, CSS 번들에 자동으로 1년 캐시를 적용하되, HTML은 `no-store`로 매번 새로 받습니다. HTML이 JS/CSS 파일의 해시 URL을 담고 있어서, HTML만 최신 상태를 유지하면 JS/CSS는 오래 캐싱해도 안전한 구조입니다.

사용자 개인 데이터는 서버 fetch 기본값을 `no-store`로 설정하고, 클라이언트에서는 React Query로 메모리 캐시를 관리했습니다. 로그아웃 시 `queryClient.clear()`와 `localStorage.clear()`를 함께 호출해 잔류 데이터가 남지 않도록 처리했습니다. 이처럼 데이터 성격에 따라 캐시 전략을 다르게 적용하는 것이 중요하다고 생각합니다.

---

### Version 3. 보안 관점형 (보안, 신뢰성을 강조하고 싶을 때)

캐시는 성능 이점이 크지만 잘못 설정하면 보안 문제와 데이터 정합성 문제를 일으킬 수 있어서, 어떤 데이터를 어디에 캐싱할지를 신중하게 설계해야 합니다.

크게 두 가지를 기준으로 생각합니다. 첫째는 데이터가 '공유 가능한가'입니다. JS, CSS처럼 모든 사용자에게 동일한 파일은 `public`으로 공격적으로 캐싱합니다. 반면 사용자 개인정보가 담긴 API 응답에 `public` 캐시를 허용하면 공용 PC나 CDN에 타인의 정보가 남을 수 있어 반드시 `private, no-store`를 사용해야 합니다.

둘째는 캐시 레이어별 책임을 명확히 하는 것입니다. HTTP 캐시는 파일 단위, React Query는 API 응답 단위로 역할이 다릅니다. 또한 인증 토큰은 절대 localStorage에 저장하지 않고, XSS로 탈취될 수 없는 HttpOnly 쿠키로만 관리합니다. 캐시를 성능 도구인 동시에 보안 설계의 일부로 바라보는 것이 중요하다고 생각합니다.
