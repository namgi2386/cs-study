# 검색

## 대표적 검색 엔진 몇가지 알아보자

### 구글의 검색 엔진을 알아보자

구글에서의 검색 결과로 나타나는 정보들은 구글이 보유한 정보/웹사이트/블로그 등이 아니다. 모두 독립된 정보들이다. 때문에 구글은 크롤링을 해서 웹사이트들의 정보를 수집해야한다. 이를 위해서 개발자는 SSE에 신경쓰며, google search advisor에 본인 사이트를 등록해야한다. 등록된 정보에 따라 구글 크롤러는 나의 사이트를 색인하고 크롤링 한 뒤, 검색 결과에 노출하게 된다. 즉, 본인 서비스 DB내에서의 data를 search, response 하는 대부분의 서비스와는 성격부터 다르다. 때문에 구글의 검색 엔진은 크롤링 > 인덱싱 > 다단계 랭킹의 파이프라인 이 사용된다. 구글 입사 준비할 것 아니라면 몰라도 되겠다.

### YouTube의 검색 엔진을 알아보자

검색 결과의 일치여부만을 판단하여 검색결과를 주는 것이 아닌 "유튜브 알고리즘"에 의해 검색 결과를 응답하는 구조이다. 때문에, 검색 결과의 멱등성을 보장하지 않는다. Transformer 모델 사용한단다.


### GitHub Code Search 검색 엔진을 알아보자

Elasticsearch로는 2억개 레포 코드를 처리하기에 느리다. 때문에 자체 개발 Blackbird (Rust) 를 사용한다.
Blackbird는 ngram/trigram/sharded ngram index 방식을 사용한다.
사용자가 git push 하면, kafka가 이벤트 발생을 전달한다. Crawler가 변경된 파일(blob)만 감지한다. Blackbird가 증분인덱싱 한다.
물론 사용자가 push한 모든 레포들이 인덱싱되는건 아니다. 트레픽없는, 검색된적도 없는 레포는 인덱싱 되지 않기도 한다. 예를들어 `repo:namgi2386/portfolio-site ProjectModal3` 검색시, `This repository's code is being indexed right now. Try again in a few minutes.` 라는 알림창이 뜬다. 이제 인덱싱해줄테니 기다리라는 뜻

### Elasticsearch / Apache Lucene (업계 표준) 를 알아보자

오픈소스 검색의 사실상 표준. Twitter, Wikipedia, Netflix, Airbnb 등이 사용.
문서의 내용을 역색인하여 인덱싱한다. 인덱스를 여러 조각으로 나누며 Shard라고 부른다. 샤드들을 여러 서버에 수평분할 하여 저장한다. 샤드 내부의 인덱스 조각은 세그먼트라 부른다. 쿼리 검색시 여러 세그먼트를 병렬 검색 후 합친다. 결과적으로 Term 사전이 만들어지며, 검색시 O(log n)으로 찾는다.

## 핵심 자료구조를 살펴보자

### 역색인

문서(단어들)를 "단어: 문서위치" 형태로 인덱싱 하여 저장하는 구조. 쿼리시, 단어 > 문서위치 를 빠르게 찾을 수 있도록 한다.
term 사전 : 역색인 결과이며, 각 단어별 posting list가 저장되며, posting list에는 문서번호(doc ID), 등장횟수(TF), 위치정보(pos)가 저장된다.

### Trie (접두사 트리)

apple, apply, apt 단어들을 ap 노드에서 분기되도록 한다. Radix Trie 압축 트라이 에서 사용한다.

### Finite State Transducer (FST)

Lucene의 Term Dictionary 구현체이며, 공통 접미사도 공유한다. 오프셋 매핑을 메모리에 올려 O(1) 조회가능하도록 한다.

## 츄츄트리는?

한번은 질문을 받은 적 있다. 츄츄트리의 문제 추천은 어떤 방식으로 구현하셨나요?
"나 : 제 나름의 알고리즘 로직으로 구현...."
좋지 않은 답변이었고, 애초에 깊은 고민이 포함되지 못한 구현방식이었다.
문제 추천도 결국 추천 결과의 랭킹을 결정해야하는 것이고, "랭킹 알고리즘"을 모른 상태로 구현 했다는게, 수준 낮은 구현 방식이었다는 뜻이다.
물론 "검색"에서의 랭킹은 사용자의 쿼리와 인덱싱된 Term 사전간 관련도를 이용하여 가장 유사한 결과를 return 하는 개념이며, 츄츄트리에서의 문제추천은 사용자 정보(JSON data)와 문제들 정보(JSON data)의 매칭이 필요하기에 다른 개념이긴 하다.
그렇다면, 문제들 정보에 문제 본문까지 포함된 상태라면, 문제 하나하나를 별개의 문서라고 판단하고, 유사도 판단을 통해 문제 추천이 가능하지 않을까 싶음.
