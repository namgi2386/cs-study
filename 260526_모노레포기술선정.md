# GitHub Organization 레포 구조 선택 근거

## 서비스 구조 개요

| 구분 | 기술 스택 | 구현 방식 |
|------|-----------|-----------|
| 백엔드 | - | 별도 레포 |
| 웹 | Next.js | 앱 웹뷰 전용 (독립 웹서비스 아님) |
| 앱 | React Native | 로그인, AR 페이지만 네이티브 구현 |

### 앱 화면 구성

| 페이지 | 구현 방식 |
|--------|-----------|
| 메인페이지 | 웹뷰 (Next.js) |
| 지도페이지 | 웹뷰 (Next.js) |
| 마이페이지 | 웹뷰 (Next.js) |
| AR페이지 | 네이티브 |
| 로그인페이지 | 네이티브 |

→ **앱 주요 화면 5개 중 3개가 Next.js에 의존**

---

## 구조 비교 (A안 vs B안)

### A안 — 레포 3개 분리

```
organization/
├── backend
├── frontend-web    (Next.js)
└── frontend-app    (React Native)
```

### B안 — 프론트 모노레포 (채택)

```
organization/
├── backend
└── frontend                    (Turborepo 모노레포)
    ├── apps/
    │   ├── web                 (Next.js 웹뷰)
    │   └── app                 (React Native)
    └── packages/
        ├── types               (공통 타입)
        ├── utils               (공통 유틸)
        └── constants           (웹뷰 URL 등 공통 상수)
```

---

## 트레이드오프 분석

| 항목 | A안 (3개 분리) | B안 (모노레포) |
|------|:---:|:---:|
| 초기 설정 난이도 | ✅ 낮음 | ❌ 높음 |
| 공통 코드 공유 | ❌ 불편 | ✅ 즉시 공유 가능 |
| 웹뷰 URL 관리 | ❌ 양쪽 중복 관리 | ✅ 단일 관리 |
| 웹-앱 인터페이스 타입 동기화 | ❌ 수동 동기화 필요 | ✅ 자동 공유 |
| 연관 작업 PR 관리 | ❌ 레포 분산 | ✅ 단일 PR 처리 |
| 배포 독립성 | ✅ 완전 독립 | ⚠️ 설정 필요 |

---

## B안 채택 핵심 근거

**1. 웹이 독립 서비스가 아닌 웹뷰 전용**
Next.js는 앱의 3개 화면을 렌더링하기 위한 용도로, 앱과 강하게 결합되어 있음. 독립 레포로 분리할 만큼의 자율성이 없음.

**2. 웹뷰 URL 단일 관리**
앱에서 웹뷰를 호출할 때 필요한 URL 상수를 `packages/constants`에서 공유하면, URL 변경 시 양쪽을 따로 수정하는 실수를 방지할 수 있음.

**3. 웹-앱 인터페이스 타입 공유**
웹뷰 ↔ 네이티브 간 `postMessage` 통신 등에서 주고받는 데이터 타입을 `packages/types`로 단일 관리. 레포 분리 시 타입 중복 정의 및 싱크 오류 위험 존재.

**4. 연관 작업의 응집성**
웹뷰 페이지 수정과 앱 WebView 컴포넌트 수정이 항상 함께 발생하므로, 단일 PR로 관리하는 것이 리뷰 및 이력 추적에 유리함.

## 예상 질문 & 답변

**Q. 모노레포를 선택한 가장 큰 이유가 무엇인가요?**

웹이 독립적인 서비스가 아닌 앱 웹뷰 전용이라는 점입니다. 앱 화면 5개 중 3개가 Next.js에 의존하고, 웹뷰 URL이나 통신 인터페이스가 바뀌면 앱 코드도 반드시 함께 바뀝니다. 이런 강한 결합 관계에서 레포를 분리하면 오히려 관리 포인트만 늘어난다고 판단했습니다.

---

**Q. 모노레포의 단점은 무엇이고, 어떻게 대응했나요?**

초기 세팅 복잡도와 CI/CD에서 변경된 앱만 선택적으로 빌드하는 설정이 필요하다는 점입니다. Turborepo를 도입해 `pipeline` 설정으로 변경된 패키지만 빌드 및 배포되도록 구성하여 이 문제를 해결했습니다.

---

**Q. 웹뷰와 네이티브 간 통신은 어떻게 처리했나요?**

`postMessage` 방식을 사용했고, 주고받는 이벤트 타입을 `packages/types`에 정의하여 웹과 앱 양쪽에서 동일한 타입을 참조하도록 했습니다. 이를 통해 타입 불일치로 인한 런타임 오류를 방지했습니다.

---

**Q. 처음부터 모노레포를 선택하지 않고 분리 레포를 선택했다면 어떤 문제가 생겼을까요?**

웹뷰 URL 변경 시 앱 코드를 별도로 수정해야 하고, 타입 정의를 양쪽 레포에서 중복 관리해야 합니다. 또한 웹과 앱을 함께 수정해야 하는 작업에서 PR이 두 레포에 분산되어 코드 리뷰와 변경 이력 추적이 어려워집니다.

---

**Q. 백엔드는 왜 모노레포에 포함하지 않았나요?**

백엔드는 프론트엔드와 공유하는 코드가 API 스펙 정도이며, 배포 주기와 기술 스택이 완전히 다릅니다. 모노레포의 이점은 강하게 결합된 코드 간에 극대화되는데, 백엔드를 포함하면 오히려 레포 복잡도만 높아진다고 판단했습니다.


## 그럼에도 A안 선택시

### A안을 선택해야 하는 상황

B안이 기술적으로 더 적합하더라도, 아래 상황에서는 A안이 합리적인 선택이 될 수 있음.

| 상황 | 이유 |
|------|------|
| **팀원 중 모노레포 경험자가 없을 때** | Turborepo 학습과 세팅에 드는 시간이 초기 개발 속도를 저해할 수 있음 |
| **웹과 앱 담당자가 명확히 분리되어 있을 때** | 각자의 레포에서 독립적으로 작업하는 것이 충돌을 줄이고 권한 관리가 명확함 |
| **웹뷰 페이지가 추후 독립 서비스로 확장될 가능성이 있을 때** | 처음부터 분리해두면 확장 시 마이그레이션 비용이 없음 |
| **CI/CD를 빠르게 단순하게 구성해야 할 때** | 모노레포의 selective build 설정 없이 각 레포를 독립 파이프라인으로 구성하는 것이 더 빠름 |
| **프로젝트 기간이 매우 짧을 때 (해커톤 등)** | 초기 세팅 비용을 최소화하고 빠른 개발에 집중해야 할 때 |

---

### A안 선택시 단점 보완 전략

A안을 선택하더라도 모노레포의 핵심 이점인 **공통 코드 공유** 문제는 반드시 해결해야 함.

**웹뷰 URL 관리**
별도의 `shared-constants` npm private 패키지를 만들거나, 웹과 앱 양쪽 레포에 `.env`로 동일한 상수를 관리하고 변경 시 양쪽을 동시에 업데이트하는 규칙을 팀 컨벤션으로 명문화.

**타입 공유**
웹뷰 ↔ 앱 간 `postMessage` 인터페이스 타입을 별도 문서(README 또는 Wiki)로 관리하고, 변경 시 반드시 양쪽 PR을 동시에 올리는 팀 규칙을 수립.

**연관 PR 추적**
GitHub Issue를 허브로 사용하여, 웹과 앱 양쪽 PR에 동일한 Issue를 연결(`closes #123`)하면 변경 이력을 한 곳에서 추적 가능.

---

### A안 선택시 예상 질문 & 답변

**Q. 웹이 웹뷰 전용인데 왜 모노레포를 선택하지 않았나요?**

팀원 4명 모두 모노레포 실무 경험이 없는 상태에서 Turborepo 세팅과 운영에 드는 학습 비용이 프로젝트 초기 개발 속도에 부담이 된다고 판단했습니다. 대신 웹뷰 URL과 통신 인터페이스 타입을 팀 컨벤션으로 명문화하여 분리 레포의 단점을 보완했고, 공유 코드가 실질적으로 많아지는 시점에 모노레포로 마이그레이션하는 점진적 전략을 선택했습니다.

---

**Q. 레포 분리 후 웹뷰 URL이나 타입이 변경될 때 싱크는 어떻게 맞췄나요?**

웹뷰 URL은 양쪽 레포의 환경변수로 관리하고, 변경 시 두 레포의 PR을 동시에 올리는 팀 규칙을 수립했습니다. 통신 인터페이스 타입은 레포 Wiki에 단일 문서로 관리하여 변경 이력을 추적했습니다. 다만 이 방식은 사람에 의존하는 구조이기 때문에, 규모가 커질 경우 모노레포로 전환하는 것이 더 적합하다는 점도 인지하고 있습니다.

---

**Q. A안을 선택한 것을 돌이켜보면 잘한 선택이었나요?**

초기 개발 속도 측면에서는 올바른 선택이었지만, 프로젝트가 진행될수록 웹과 앱의 연관 작업에서 PR이 분산되고 타입 싱크를 수동으로 맞추는 불편함이 실제로 발생했습니다. 이 경험을 통해 웹뷰 의존도가 높은 구조에서는 초기부터 모노레포로 가는 것이 장기적으로 더 효율적이라는 것을 체감했고, 다음 프로젝트에서는 B안을 선택할 것 같습니다.
